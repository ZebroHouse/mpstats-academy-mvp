# Сквозной AI-ассистент MAAL — дизайн

**Дата:** 2026-07-08
**Статус:** Design approved, ждёт имплементационного плана
**Ветка:** `feature/ai-assistant`

## 1. Проблема и цель

Половина зарегистрированных юзеров не открывает ни одного урока. У платформы уже есть
мощная RAG-машинерия (чат в уроке, семантический поиск, движок «намерение→задачи»), но она
разрознена и привязана к контексту конкретного урока. Нет единой точки входа, где юзер мог бы
одним вопросом получить и ответ по существу, и маршрут к нужному контенту.

**Цель:** сквозной AI-помощник — компаньон, живущий на всех страницах платформы, который:
- отвечает на вопросы по материалам академии и смежным селлерским темам (RAG + знания LLM в домене);
- подбирает релевантные уроки и задачи-плейбуки, подмешивая их карточками прямо в ответ;
- даёт юзеру добавить их в избранное в один клик;
- работает как воронка: бесплатный юзер, поболтав, получает рекомендации платного контента.

## 2. Позиционирование (гибрид с приоритетом навигации)

Агент по умолчанию ведёт себя как навигатор по платформе, но на селлерские и смежные
бизнес-вопросы отвечает по существу и мостит к контенту. Спектр «RAG vs общие знания»
зафиксирован как **RAG-first + общий фолбэк только в домене**:

- Есть релевантные чанки в RAG → отвечает из них.
- Нет → отвечает общими знаниями LLM, **но только если вопрос в домене**.
- Вне домена → отказ.

### Граница домена — «кольца»

Домен определяется не темой, а работой юзера: «веду и растю бизнес на маркетплейсах».

| Кольцо | Примеры | Поведение |
|---|---|---|
| **Ядро** | карточки, реклама, аналитика, выкупы, рейтинг, поставки WB/Ozon | Отвечаем, приоритет RAG |
| **Смежное (in-domain)** | финансы бизнеса (PnL, юнит-экономика, ДРР, кэшфлоу), налоги ИП/самозанятого, продвижение/маркетинг, операционка, логистика, найм | Отвечаем по существу → мост к урокам |
| **Снаружи (отказ)** | код, школьные задачи, медицина, политика, творчество, **личные** финансы (ипотека, вклады), написание текстов не про МП-бизнес | Вежливый отказ + возврат к теме |

**Принцип калибровки:** ошибка «ответил на пограничный бизнес-вопрос» дешевле ошибки «отказал
реальному селлеру». Кольцо смежного держим щедрым; жёстко режем только явно-снаружи.

**Ключевое различение против абьюза-через-финансы:** бизнес-финансы селлера = да; личные/общие
финансы = нет. «Из чего складывается ДРР» → да. «Посчитай мне ипотеку» → нет.

### Класс «дай живой ответ / выбери за меня»

Вопросы вроде «выбери мне нишу на миллион в 2026» требуют живых рыночных данных (в RAG их нет,
LLM их выдумает) и директивного финсовета (репутационный/юридический риск). Правило:

> **Агент учит методу и роутит в уроки/сервис MPSTATS; он не выдумывает живые рыночные данные
> и не даёт директивных «вложи сюда» советов.**

Пример отклика: объясняет *как* выбирают нишу под бюджет (спрос/конкуренция/маржа/сезонность),
даёт урок «Поиск ниши» + «Юнит-экономика», и мостит в сам сервис MPSTATS для проверки гипотез
на живых данных. Тот же принцип покрывает «посчитай мою точную прибыль», «гарантированно ли
я заработаю» и подобное.

## 3. Архитектура — детерминированный RAG-пайплайн (НЕ tool-loop)

Ключевое решение по надёжности/стоимости: под капотом «агент» — это фиксированный конвейер,
переиспользующий существующие примитивы. Никакого открытого ReAct-цикла в v1.

```
Сообщение юзера
  │
  ├─ 1. Домен-гейт (дешёвый вызов mini: «про бизнес продавца?»)
  │      └─ нет → канон-отказ. Ретрив НЕ трогаем. Квоту НЕ тратим. STOP.
  │
  ├─ 2. Расширение запроса: expandSellerQuery (аббревиатуры) + эмбеддинг
  │
  ├─ 3. Ретрив параллельно:
  │      • чанки уроков — машинерия searchLessons (packages/ai retrieval)
  │      • задачи-плейбуки — машинерия intent.resolve (job embeddings + chunk aggregation)
  │
  ├─ 4. Синтез (mini): { answer, lessonIds[], jobIds[] }
  │      └─ lessonIds/jobIds — ТОЛЬКО из кандидатов ретрива (whitelist, anti-hallucination)
  │
  └─ 5. Обогащение (сервер): id → данные карточек (title/slug/duration/lessonCount)
```

**Переиспользуемые компоненты (уже в кодовой базе):**
- `packages/ai/src/seller-lexicon.ts` — `expandSellerQuery`
- `packages/ai/src/retrieval.ts` + `packages/api/src/routers/ai.ts` — семантический ретрив уроков
- `packages/ai/src/intent/` — ретрив и обогащение задач-плейбуков
- `packages/api/src/routers/favorite.ts` — запись в избранное (через UI, не агентом)
- `apps/web/src/lib/rate-limit.ts` — лимитирование (паттерн партнёрской ручки)
- `LessonCard` / `JobCard` — существующие карточки с сердечком-избранным (после витрины)

Реальный tool-calling loop и живые данные из MPSTATS API — явно на будущее.

## 4. Guardrails

- **Домен-гейт как отдельный вход-фильтр**, а не только инструкция в системном промпте
  (системный промпт джейлбрейкают; дешёвая классификация перед ретривом надёжнее).
- **Anti-hallucination whitelist:** модель может ссылаться только на id из кандидатов ретрива.
- **Дисклеймер** в подвале панели: «Помощник отвечает по материалам академии. Не финансовый совет.»
- Правило «метод, не выдумка живых данных» (см. §2).

## 5. Действия: агент предлагает, юзер жмёт

- Все инструменты агента **read-only** (`searchLessons`, `intent.resolve`).
- Единственная запись — сердечко на карточке в ответе → `favorite.add`, инициирует юзер кликом.
- Агент **не может** намутить в аккаунте юзера сам (нет авто-действий).
- Формат ответа модели: `{ answer: string, lessonIds: string[], jobIds: string[] }`.
  Сервер обогащает id → карточки (как это уже делает `intent.resolve` для recommend).

## 6. Доступ и лимиты

Доступ делим по существующему `access.ts` (наличие активного доступа), новую сущность не заводим.

| Тир | Доступ | Дневной лимит |
|---|---|---|
| TRIAL (активный) + ACTIVE-подписка | Полный | **50 сообщений/день** |
| Free (нет активного доступа) | Квота | **5 сообщений/день** |

- Триал видит ровно то же, что платный — чтобы понимать, за что платит.
- Поверх дневного у **всех** — минутный анти-скрипт **6 сообщений/мин** (человеку невидим,
  режет ботов/шаринг аккаунта).
- **Офф-топик-отказ квоту НЕ тратит** (гейт отбивает до ретрива; дешёвый, UX добрее).
- Сброс дневной квоты — в полночь **МСК**.
- **Экран упора у фришника** — не сухое «лимит исчерпан», а конверсионное касание:
  «Бесплатные вопросы на сегодня закончились. На подписке помощник без ограничений — [оформить]».
- Незалогиненным доступа нет (нужна и персистенция, и защита от абьюза).

Себестоимость: ~полцента за сообщение на gpt-4.1-mini (≈8k вход + 1k выход, 2 вызова).
Квота — рычаг конверсии, не экономии.

## 7. Персистенция

- **Одна непрерывная нить на юзера, без веток.** Ветки — неверный ментальный образ для
  компаньона обучения; один тред = «мой помощник меня помнит».
- В контекст модели передаём **последние ~10 сообщений**; полная история — в БД.
- Полная история даёт аналитику «о чём спрашивают» и материал для будущего дообучения RAG.
- **Кнопка «Новый разговор»** в шапке панели — архивирует текущий тред и начинает с чистого
  листа (это сброс, не ветка).
- Суммаризация длинной истории — за скоупом v1 (пока просто последние N).

### Модель данных (новые аддитивные таблицы)

```prisma
model AssistantConversation {
  id        String   @id @default(cuid())
  userId    String
  status    String   @default("active")   // active | archived
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt
  messages  AssistantMessage[]
  @@index([userId, status])
}

model AssistantMessage {
  id             String   @id @default(cuid())
  conversationId String
  role           String   // user | assistant
  content        String   @db.Text
  lessonIds      String[] // подмешанные карточки уроков (для assistant-сообщений)
  jobIds         String[] // подмешанные карточки задач
  inDomain       Boolean  @default(true) // false = отказ (для аналитики)
  createdAt      DateTime @default(now())
  conversation   AssistantConversation @relation(fields: [conversationId], references: [id], onDelete: Cascade)
  @@index([conversationId, createdAt])
}
```

Миграция — аддитивная, применяется к prod-Supabase через Mgmt API (паттерн проекта).
Дневной/минутный счётчик квоты — в rate-limit слое (не обязательно в этих таблицах).

## 8. UI

- Кнопка-пилюля **«AI-ассистент»** (иконка-искра) в верхней панели рядом с
  уведомлениями/онбордингом/профилем.
- Клик → **правый drawer во всю высоту** (≈450px). Свой крестик; повторный клик по кнопке
  закрывает. Компаньон на всех страницах (в `(main)/layout.tsx`), отдельной страницы нет.
- Внутри drawer: диалог + подмешанные `LessonCard`/`JobCard` (реюз) + инпут + дисклеймер +
  плашка остатка бесплатных вопросов (для free) + кнопка «Новый разговор».
- Мобайл — нижний лист (bottom sheet ~82vh).
- Референс-макет: `scratchpad/ai-assistant-mockup.html` (вариант B выбран owner из A/B/C).

## 9. Модель и отдача

- **gpt-4.1-mini** на всё (гейт + синтез). $0.40/$1.60 за 1M вход/выход (OpenRouter, 08.07.2026).
  Качество держится на RAG-грунте и правилах поведения, а не на силе модели.
- **Ответ стримим** токен-за-токеном (молча ждать 3–8 сек флагманский помощник не должен);
  карточки прилетают в конце. Если стриминг через текущий tRPC-стек дорог по времени — фолбэк
  на non-streaming + индикатор печати.
- Апгрейд-путь при слабом качестве: расщепить — дешёвый mini на гейт+роутинг, модель посильнее
  только на финальный синтез. Оптимизация под факт, не в v1.

## 10. Явно ЗА скоупом v1 (YAGNI)

- Ветки чатов.
- Живые данные из MPSTATS API (tool call в сервис).
- Настоящий tool-calling / ReAct-loop.
- Доступ незалогиненным (лид-магнит на лендинге — отдельно).
- Авто-действия агента (только propose→click).
- Суммаризация длинной истории.
- Дообучение RAG на собранных диалогах (данные копим, обучение — потом).
