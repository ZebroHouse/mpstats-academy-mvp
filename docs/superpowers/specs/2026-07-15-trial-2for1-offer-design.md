# Акция «2 месяца по цене одного» для триал-юзеров — design spec

**Дата:** 2026-07-15
**Ветка:** `feature/trial-2for1-offer`
**Источники:** ТЗ (Google Doc 09.07.2026) + прототип `pricing_prototype (3).html`
**База:** прод `762153f` (master), поверх выкатанной discount-инфры (v1.29, `2884a7f`) и AI-ассистента.

## 1. Цель и суть

Дать юзерам с **активным триалом** одноразовое спецпредложение: первый платёж **2 990 ₽ открывает полный доступ (PLATFORM) на 2 месяца**, дальше стандартное автосписание **2 990 ₽/мес**. Предложение живёт только внутри окна акции; вне окна работает обычная модель.

**Ключевой принцип — оффер это параллельная сущность, не замена базовой модели.** Базовый PLATFORM 2990/30д существует всегда. Оффер — альтернативный путь *первой покупки*, доступный только eligible-триал-юзеру.

Три модуля: **A** — движок оффера (backend + платёж), **B** — sticky-баннер, **C** — новая страница тарифов. Общий контракт между B и C — эндпоинт `offer.getState`.

## 2. Что берём из ТЗ и как адаптируем

| Из ТЗ | Решение на нашем продукте |
|---|---|
| Оффер «2 мес по цене 1» (2990 → 60д → 2990/мес) | Модификатор первого периода поверх существующей CP-инфры (см. §3). |
| Один раз на человека, проверка на сервере | `OfferRedemption` (@@unique userId), резолвер серверный. |
| Действует триал + сутки grace | Окно = trial активен ИЛИ ≤24ч после конца триала. |
| Баннер, состояние «активировать триал» | **Выкинуто** — триал выдаётся автоматически при регистрации (`ensureBaseTrial`), состояния «не активирован» не существует. |
| Sticky-баннер, 3 состояния по статусу | 2 оффер-состояния (trial_active, grace) + expired-апселл + скрыт (см. §4). |
| Новая страница тарифов по прототипу | Порт секций прототипа в наш v8-дизайн (см. §5). |

## 3. Модуль A — движок оффера

### 3.1. Механика (подтверждено)

- Первый платёж **2 990 ₽** → полный доступ PLATFORM на **60 дней от момента оплаты**.
- Автосписание **2 990 ₽ / 30 дней**, начиная с **60-го дня**.
- Применяется **автоматически** по статусу триала (без ввода промокода).
- Только тариф **PLATFORM**. COURSE (1 990) — всегда обычный.

### 3.2. Окно eligibility (once-per-person)

Пользователь eligible, если ВСЕ верно:
1. Есть подписка `status=TRIAL` с `currentPeriodEnd > now` **ИЛИ** `now <= currentPeriodEnd + 24ч` (grace).
2. Нет активной платной PLATFORM-подписки.
3. Оффер ещё **не использован** (нет `OfferRedemption` для этого userId).

Проверка **только на сервере** (ТЗ: «срок проверяется на нашей стороне, не в браузере»). Повторный триал невозможен by design (`ensureBaseTrial` = «нет подписок вообще»), значит повторно попасть в окно нельзя. Юзерам, чей триал истёк >24ч к моменту запуска, оффер **не** показываем (никакой отдельной догоняющей рассылки).

### 3.3. Архитектура — «модификатор первого периода» (выбранный подход A)

Оффер это НЕ скидка суммы и НЕ отдельный план. Это флаг «первый период = 60 дней» + отложенный рекуррент.

**Новое:**
- `resolveApplicableOffer(userId, plan)` в `packages/api/src/services/` — по образцу `resolveApplicableDiscount`. Возвращает `{ firstPeriodDays: 60, recurrentStartDate: <now+60d> } | null`. Только PLATFORM + eligible.
- Модель `OfferRedemption { id, userId @unique, subscriptionId, offerKey, redeemedAt }` — consume-once (пишется при успешной первой оплате оффера).
- Поле на `Subscription`: `offerFirstPeriodDays Int?` — несёт длину первого периода на pending-подписке (60), чтобы вебхук её уважил.

**Правки существующего:**
- `billing.initiatePayment` — вызвать `resolveApplicableOffer`; если оффер применим, записать `offerFirstPeriodDays=60` на pending-подписку и вернуть `recurrentStartDate` для виджета. Сумма первого платежа = `plan.price` (2990, не трогаем `firstAmount`). Рекуррент пиннится к `plan.price` (уже так).
- `openPaymentWidget` — уже поддерживает `recurrent.startDate` (проверено, `widget.ts:51`). Передаём `recurrentStartDate`.
- `handlePaymentSuccess` (вебхук, `subscription-service.ts`) — сейчас жёстко `currentPeriodEnd = now + plan.intervalDays`. Правка: **если `sub.offerFirstPeriodDays` задан — использовать его** для первого периода вместо `plan.intervalDays`. Записать `OfferRedemption`. Рекуррентные продления (`decide-recurrent-update.ts`) остаются на `plan.intervalDays` (30) — не трогаем.

**54-ФЗ чеки:** переиспользуем существующий раздельный паттерн (first vs recurrent receipt). Первый чек — 2990 (описание позиции = доступ PLATFORM на 60 дней). Рекуррентный чек — 2990 на полную цену (линчпин уже пройден картовым тестом на v1.29).

### 3.4. Взаимодействие с discount-кодами

Оффер (модификатор периода) и discount-коды (скидка суммы) — ортогональны. MVP-правило: **если применён discount-код, оффер не накладывается** (не удлиняем период у уже сниженной суммы). Резолвер оффера возвращает `null`, когда для юзера/сессии резолвится discount. Приоритет и крайние случаи финализируются на этапе плана.

## 4. Модуль B — sticky-баннер

### 4.1. Размещение

- Живёт в **том же слоте**, где сейчас реферальный баннер — `apps/web/src/app/(main)/layout.tsx` (над шапкой, в колонке контента справа от сайдбара).
- **Но sticky** — прибит к верху, не уезжает при скролле (реф-баннер уезжал). Согласованный z-index со sticky-шапкой (`header` = `z-40`; баннер выше, шапка ниже, либо баннер над шапкой в общем sticky-стеке).
- **Реферальный баннер снимается с монтирования целиком** (эксперимент «поживём без него» — он малорезультативен). Компонент `ReferralBanner.tsx` из кода **не удаляем** — просто не рендерим, чтобы легко вернуть. Реф-программа на `/profile/referral` не трогается.
- Решение о показе — **серверное**, в `(main)`-layout (он уже серверный и уже читает `prisma.subscription`). Layout считает состояние через `resolveApplicableOffer`/`offer.getState` и рендерит `OfferBanner` (или ничего).

### 4.2. Состояния

| state | Условие | Контент | Таймер | CTA |
|---|---|---|---|---|
| `trial_active` | триал активен | «2 месяца по цене одного» + «До конца бесплатного доступа · дальше 2 990 ₽/мес» | живой, до `trialEnd` | «Забрать предложение» → /pricing |
| `grace` | ≤24ч после триала | то же, «Предложение действует ещё…» | живой, до `trialEnd+24ч` | «Забрать предложение» |
| `expired` | >24ч, оффер закрыт | «Откройте все уроки платформы» | нет | «Смотреть тарифы» → /pricing |
| — (скрыт) | оплачено / оффер использован / dismissed | баннер не рендерится | — | — |

### 4.3. Компонент

- `OfferBanner` (`'use client'`) — client только ради тикающего таймера. Получает `state` + `endsAt` пропами от сервера (лейаута). `setInterval` 1с, честный отсчёт, `tabular-nums`. Формат: `Nд HH:MM` пока >1ч, `MM:SS` под конец.
- **Визуал — Вариант 1 «Dark premium»:** градиент `mp-blue-900 → mp-blue-800`, чип таймера `mp-pink-500` (белый текст), белая CTA-кнопка (текст `mp-blue-700`), иконка 🎁. Мокап-референс: `scratchpad/offer-banner-mockups.html` (вариант 1).
- **Dismiss (✕)** → `sessionStorage` (свернуть «до следующего захода» = до новой сессии; ТЗ). НЕ localStorage.
- **Источник правды** для таймера/состояния — сервер. Клиент не пересчитывает окно.

### 4.4. Эндпоинт-контракт

`offer.getState` (protectedProcedure) → `{ state: 'trial_active'|'grace'|'expired'|'none', offerEndsAt: string|null, priceLabel: {...} }`. Переиспользует `resolveApplicableOffer`. Используется и баннером (модуль B), и страницей тарифов (модуль C) — единый источник.

## 5. Модуль C — страница тарифов

Порт секций прототипа в наш **v8-дизайн** (переиспользуем `V8Header`/`V8Footer`/`Reveal`/`StickyCTA`), светлый макет прототипа перекладываем в дизайн-систему продукта. Заменяет текущую `apps/web/src/app/pricing/page.tsx`.

Секции сверху вниз:
1. **Полоса-оффер с таймером** — та же `offer.getState`; для eligible-триала показывает отсчёт, иначе скрыта.
2. **Hero** — заголовок.
3. **Отзывы бегущей строкой** — 11 реальных цитат (уже в прототипе), инициалы-кружки сейчас, фото добавляются позже. Pause-on-hover, `prefers-reduced-motion` фолбэк (без анимации, wrap).
4. **Два тарифа:**
   - COURSE **1 990 ₽** (пикер курса).
   - PLATFORM **2 990 ₽** (выделен). Для eligible-триала — оффер-режим: зачёркнутая **5 980 ₽** + «**Первые 2 месяца — 2 990 ₽, далее 2 990 ₽/мес**» + note «до конца бесплатного доступа». Иначе — стандартная формулировка. Режим из `offer.getState`.
5. **Что входит** — value-карточки.
6. **FAQ** — включая «Как отменить подписку» (cancel-текст живёт **только** здесь).
7. **Промокод** — существующая обвязка `promo.activate`/auto-`?promo=` без изменений.
8. **Sticky mobile CTA**.

**Редполитика ([[feedback_no_cancel_language_top_of_funnel]]):** убрать «Отмена в любой момент» из карточки/футера/hero; trust-строка карточки в reminder-формулировке («Напомним письмом за 3 дня до списания»). Финальный проход копирайта — скиллом `mpstats-copywriting`.

**Цены:** полная формулировка без сокращений везде (ТЗ). Числа тарифов — из `billing.getPlans`; счётчики курсов/уроков — актуальные (не placeholder `[N]`).

## 6. Данные и поток

```
Триал-юзер → /pricing (или CTA баннера)
  page.tsx → offer.getState → оффер-режим карточки PLATFORM
  «Открыть всё» → billing.initiatePayment
     → resolveApplicableOffer(user, PLATFORM) = {firstPeriodDays:60, recurrentStartDate:+60d}
     → pending Subscription { offerFirstPeriodDays:60 }, amount=2990
     → openPaymentWidget({ amount:2990, recurrent:{interval:'Month',period:1,startDate:+60d}, receipt, recurrentReceipt })
  CP «pay» webhook → handlePaymentSuccess
     → currentPeriodEnd = now + offerFirstPeriodDays(60)   // вместо plan.intervalDays
     → OfferRedemption(userId) written
  CP recurrent (день 60) → decide-recurrent-update → +plan.intervalDays(30) как обычно
```

## 7. Edge cases / error handling

- **Триал истёк между показом и оплатой (>grace):** резолвер на `initiatePayment` перепроверяет окно — если вышел, оффер не применяется, юзер платит обычные 2990/30д (без сюрприза; UI подстроится по `getState`).
- **Двойной клик/повторная оплата:** `OfferRedemption @@unique(userId)` + существующая идемпотентность вебхука по `InvoiceId`.
- **Оффер + discount-код одновременно:** discount выигрывает, оффер `null` (§3.4).
- **Юзер уже платный / оффер использован:** `getState=none`, баннер скрыт, карточка стандартная.
- **Отмена в grace-периоде:** стандартный `billing.cancelSubscription`.
- **Timezone/честность таймера:** сервер отдаёт `offerEndsAt` ISO; клиент тикает локально, но окно валидируется сервером на оплате.

## 8. Тестирование

- **Unit:** `resolveApplicableOffer` (все ветки eligibility), webhook first-period override (60 vs 30), OfferRedemption consume-once, discount-vs-offer приоритет.
- **Инвариант:** триал-строка не мутируется (существующий `trial-invariant.test.ts`), оффер создаёт отдельную ACTIVE-подписку.
- **E2E/ручной:** через `/admin/billing-test` (hidden test plan) — картовый прогон: первый платёж 2990, `currentPeriodEnd = +60д`, рекуррент запланирован на день 60 с `amount=2990`. **UAT-линчпин:** проверить в дашборде CloudPayments, что recurrent.amount = полная цена (2990), а первый период = 60 дней.
- Баннер: 3 состояния + скрытие, dismiss-per-session, таймер тикает.

## 9. Фазы реализации

1. **Фаза A — движок оффера** (schema `OfferRedemption`+`offerFirstPeriodDays`, `resolveApplicableOffer`, `initiatePayment`, webhook override, `offer.getState`, тесты). Проверяется через `/admin/billing-test` до всякого UI.
2. **Фаза B — баннер** (`OfferBanner` Вариант 1, монтаж в `(main)`-layout sticky, снятие `ReferralBanner`, dismiss-per-session).
3. **Фаза C — страница тарифов** (порт секций в v8, оффер-режим карточки, отзывы-marquee, FAQ+редполитика, копирайт-проход).

Общий контракт `offer.getState` — между A, B, C.
