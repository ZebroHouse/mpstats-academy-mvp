## 1. Зачем

Сегодня раздел «Обучение» — это одна страница (`/learn`), которая пытается делать три разные работы одновременно: вести по плану после диагностики, давать быстрый ответ на конкретный вопрос и решать бизнес-задачу. Из-за этого ни один сценарий не работает хорошо: счётчики не сходятся между экранами, «персональный план» выглядит как стена уроков, пользователь не понимает, с чего начать.

Редизайн разводит «Обучение» на **четыре сущности с ясной зоной ответственности** + перестраивает Главную в **точку входа в три сценария**. Плюс чинит поиск (контекстный, по сущности раздела) и выводит его на видное место.

**Связь с задачами owner:**
1. ~~Диагностика на WB/Ozon~~ — **сделана** (Phase 59 v2), вне этого спека.
2. **Починить поиск, развести по сущностям** → раздел [4. Контекстный поиск](#4-контекстный-поиск-задача-2).
3. **Развести «Обучение» на сабменю** → раздел [3. Четыре сущности](#3-четыре-сущности-задача-3).
4. **UI: дашборд + поиск крупно** → раздел [5. UI](#5-ui-дашборд--hero-поиск-задача-4).

**Метрики (из презентации):** Activation ↑ (понятный первый шаг), Retention M1 ↑ (виден прогресс), WAU/MAU ↑ (Избранное = повод возвращаться).

## 2. Терминологический канон

Убираем англицизмы. Фиксируем язык интерфейса и команды:

| БЫЛО (в продукте) | СТАЛО | Что это |
|---|---|---|
| Трек / Мой трек | **Персональный план** | индивидуальный маршрут после диагностики |
| Плейбук / джоба (Job) | **Решение под задачу** / Задача | инструкция под бизнес-задачу, не курс |
| Каталог / Все курсы | **База знаний** | общее хранилище контента (уроки + материалы) с поиском |
| «+ В трек» (ручное) | **«В избранное»** (сердечко) | личная подборка пользователя |

> **Важно:** ренейминг затрагивает только пользовательские строки и URL. Имена моделей БД (`Job`, `LearningPath`) **не переименовываем** — это внутренний код, ренейм там = лишний риск без пользы. Канон применяется к UI-строкам и маршрутам.

## 3. Четыре сущности (задача 3)

«Обучение» в боковом меню становится раскрывающимся пунктом с 4 под-разделами:

| Под-раздел | URL | На базе чего (AS-IS) | Режим пользователя |
|---|---|---|---|
| **Персональный план** | `/learn/plan` | `LearningPath` (нынешний `/learn/track`), но **только диагностическая часть** | «учусь системно» |
| **Решения под задачу** | `/learn/solutions` | `Job`-каталог (нынешний lens «По задачам») | «решаю задачу» |
| **База знаний** | `/learn/library` | `Course`/`Lesson` (нынешний lens «Все курсы») + материалы | «ищу ответ» |
| **Избранное** | `/learn/favorites` | **новая модель `Favorite`** | «сохранил, чтобы вернуться» |

Нынешний `/learn` с переключателем `lens` (`jobs`/`courses`) **расщепляется** на отдельные страницы. `/learn` → редирект на дефолтный под-раздел (Персональный план, если есть трек; иначе База знаний).

### 3.1 Персональный план (`/learn/plan`)

- Источник: `getRecommendedPath` — **только** диагностические секции (по 5 компетенциям). Ручные добавления убраны (см. [6. Разведение План/Избранное](#6-разведение-планизбранное)).
- Содержит: джобы, рекомендованные диагностикой, + уроки маршрута.
- Пустое состояние: CTA «Пройти диагностику».

### 3.2 Решения под задачу (`/learn/solutions`)

- Источник: `job.getCatalog` (как сейчас lens «По задачам»), `MarketplaceSwitch` (WB/Ozon), фильтр по прогрессу.
- Карточка решения: `/learn/job/[slug]` (без изменений маршрута).
- Действие на решении: «В план» (`addJobToTrack` — добавляет в трек) **или** «В избранное» — см. [6](#6-разведение-планизбранное).

### 3.3 База знаний (`/learn/library`)

- Источник: `learning.getCourses` (аккордеон курсов/уроков, как нынешний lens «Все курсы») + **материалы** (см. [3.4](#34-материалы-в-базе-знаний)).
- Открыта **без диагностики** (это утилита «у меня вопрос прямо сейчас»).
- Курсы становятся здесь **фильтром**, а не отдельной навигацией.

### 3.4 Материалы в Базе знаний

Сейчас `Material` (типы: `PRESENTATION`, `CALCULATION_TABLE`, `EXTERNAL_SERVICE`, `CHECKLIST`, `MEMO`) — только admin-управляемые, привязаны к урокам (`LessonMaterial`), у юзера видны только внутри урока (`LessonMaterials.tsx` → `material.getSignedUrl`). Флаг `Material.isStandalone` существует с пометкой *«D-04: задел под Library, в этой фазе не используется»* — это и есть наш задел.

**Делаем:**
- Новый user-facing эндпоинт `material.listForUser` (protected) — отдаёт видимые материалы (`isHidden=false`), с фильтром по типу и поиском по `title`. Включает standalone-материалы + материалы, привязанные к доступным урокам.
- В Базе знаний — раздел/фильтр материалов (по типу). Карточка — `MaterialCard.tsx` (уже есть).
- ACL на скачивание — без изменений (`getSignedUrl` уже проверяет подписку).

### 3.5 Избранное (`/learn/favorites`)

- Новая модель `Favorite` (см. [7. Модель данных](#7-модель-данных-favorite)).
- Содержит: уроки, решения (джобы) **и** материалы, которые юзер сам сохранил сердечком.
- Фильтр по типу (Уроки / Решения / Материалы) — на базе `Favorite.itemType`.
- «Мои подборки» (коллекции со слайда 15) — **вне scope этого захода** (YAGNI; добавим, когда понадобится группировка). Фиксируем как future idea.

## 4. Контекстный поиск (задача 2)

**Проблема:** `AgentSearch` (маленькая строка h-12 на `/learn`) всегда зовёт `intent.resolve({surface:'learn'})` и **всегда возвращает плейбуки** — даже в lens «Все курсы», где юзер ждёт уроки.

**Решение:** поиск привязан к разделу, ищет сущность этого раздела:

| Раздел | Что ищет | Бэкенд |
|---|---|---|
| Решения под задачу | плейбуки (джобы) | `intent.resolve` (как сейчас) |
| База знаний | уроки + материалы | `ai.searchLessons` (**уже существует, не подключён к UI**) + `material.listForUser` поиск |
| Избранное | сохранённое юзером | локальный фильтр по `Favorite` |

- `ai.searchLessons` — семантический поиск по урокам, готов в `packages/api/src/routers/ai.ts:213`, сейчас не используется ни одним компонентом. Подключаем в Базе знаний.
- Поиск материалов — `contains` по `title` (как в admin `material.list`), без семантики на этом заходе.
- `AgentSearch` рефакторим: принимает проп `scope: 'solutions' | 'library'` и роутит на нужный бэкенд + рендерит нужный тип карточки результата.

## 5. UI: дашборд + hero-поиск (задача 4)

### 5.1 Дашборд → 3 входа (слайд 17)

Сейчас `/dashboard`: 4 голых счётчика (уроков / минут / streak / %) + 2 карточки-CTA + радар + лента. Для нового юзера 4 нуля = «мёртвая платформа».

**Делаем** — 3 крупных акцентных входа вверху:
| Вход | Ведёт | Подпись |
|---|---|---|
| **Продолжить мой план** | `/learn/plan` | к месту, где остановился |
| **Найти быстрый ответ** | `/learn/library` | поиск по урокам и материалам |
| **Решить задачу** | `/learn/solutions` | каталог инструкций под задачи |

- Счётчики ужимаем (из 4 → компактная строка, нулей у нового юзера быть не должно).
- Радар навыков, «Продолжить урок», лента активности — остаются (ниже входов).
- *Комьюнити-блок «Жизнь Академии» (слайд 36, направление 06) — отдельное направление, вне scope этого спека.*

### 5.2 Hero-поиск в разделах обучения (слайд 6)

- Крупный hero-блок с поисковой строкой **вверху** разделов «Решения под задачу» и «База знаний» (вместо нынешней мелкой h-12 строки).
- Больше экранного пространства, акцентный фон, фильтры-чипы под строкой.

## 6. Разведение План/Избранное

**Решение owner:** Персональный план = диагностика; Избранное = ручные добавления.

AS-IS: `LearningPath` смешивает:
- **диагностические** уроки — секции по компетенциям (`getRecommendedPath` строит из диагностики);
- **ручные** добавления — секция `custom` («Мои уроки», через `addToTrack`) + `addedJobs[]` (джобы через «+ В трек»).

TO-BE:
- **Персональный план** = только диагностические секции.
- **Избранное** = `custom`-секция (ручные уроки) + `addedJobs[]` (ручные джобы) → переезжают в `Favorite`.

**Миграция (~170 юзеров, без потери данных — переклассификация):**
- Для каждого `LearningPath`: уроки из `custom`-секции → `Favorite(itemType=LESSON)`; элементы `addedJobs[]` → `Favorite(itemType=JOB)`.
- `LearningPath.addedJobs` после миграции не используется фронтом Плана (оставляем колонку dormant либо чистим — решим на этапе планирования фазы D).
- `LessonProgress` **не трогаем** (D-07-стиль hard rule из Phase 58): прогресс просмотра остаётся независимо от того, в Плане урок или в Избранном.

> **Открытый вопрос для фазы D:** различает ли `getRecommendedPath` сейчас секцию `custom` от диагностических надёжно (по `section.id === 'custom'`)? Проверить на этапе планирования — от этого зависит точность миграции. (Из `/learn/page.tsx:124` видно, что `custom`-секция помечается `id: 'custom'`, title «Мои уроки» — признак есть.)

## 7. Модель данных: `Favorite`

Новая модель (additive-миграция, безопасно для prod):

```prisma
enum FavoriteItemType {
  LESSON
  JOB
  MATERIAL
}

model Favorite {
  id        String           @id @default(cuid())
  userId    String
  itemType  FavoriteItemType
  itemId    String           // Lesson.id | Job.id | Material.id
  createdAt DateTime         @default(now())

  user UserProfile @relation(fields: [userId], references: [id], onDelete: Cascade)

  @@unique([userId, itemType, itemId])
  @@index([userId, itemType])
}
```

- Полиморфная ссылка через `(itemType, itemId)` — без FK на три таблицы (Prisma не поддерживает полиморфные FK; целостность на уровне приложения).
- tRPC: `favorite.add`, `favorite.remove`, `favorite.list` (с фильтром по типу), `favorite.isFavorited` (батч для проставления состояния сердечек в каталогах).
- Кнопка-сердечко — общий компонент `FavoriteButton`, на карточках урока/решения/материала.

## 8. Декомпозиция на фазы

Единый редизайн, **разработка пофазно** (атомарные коммиты), **деплой — единый**: всё собирается на ветке `learning-2.0-redesign` → staging → QA → правки → merge в master → прод. Master не трогаем до проверки на стейдже (`feedback_deploy_flow_control.md`).

| Фаза | Содержание | Зона | Риск | Зависит от |
|---|---|---|---|---|
| **A. Каркас + ренейминг** | Сабменю «Обучение» (4 под-пункта); routes `/learn/plan` `/solutions` `/library` `/favorites`; расщепление lens на страницы; ренейминг UI-строк по канону | UI + навигация | низкий | — |
| **B. Контекстный поиск** | `AgentSearch` с `scope`; Решения→`intent.resolve`, База знаний→`ai.searchLessons`; разные карточки результатов | API + UI | средний | A |
| **C. Материалы в Базе знаний** | `material.listForUser`, включить `isStandalone`, каталог материалов + фильтр по типу + поиск | API + UI | средний | A, B |
| **D. Избранное** | Модель `Favorite` + миграция; `FavoriteButton` (сердечко); переезд ручных добавлений из трека; План→чисто диагностический | DB + API + UI | **высокий** | A, C |
| **E. UI: дашборд + hero-поиск** | Дашборд 3 входа (слайд 17), счётчики ужать; hero-блок поиска в разделах (слайд 6) | UI | низкий | A, B |

Порядок разработки: **A → B → C → D → E**.

## 10. Совместимость и риски

- **Старые ссылки `/learn/track`** — оставить редирект на `/learn/plan` (внутренние ссылки + возможные закладки/письма).
- **`addedJobs[]` / `addToTrack`** — поведение меняется (см. фаза D); проверить все вызовы (`AgentSearch`, `JobCard`, `/learn` курсы) на этапе планирования D.
- **Миграция трек→избранное** — высокий риск, делать additive + idempotent; backup перед запуском (PROD DATABASE SAFETY).
- **Tour / `data-tour` якоря** (`learn-view-toggle`, `learn-search`, `learn-add-to-track`) — обновить под новую структуру, иначе онбординг-тур сломается.
- **CarrotQuest события** — при изменении флоу проверить `pa_*` события трека/диагностики.
