# Дизайн: Бесшовный вход из MPSTATS на платформу (Часть 2)

**Дата:** 2026-06-10 (ревизия 2026-06-11 после уточнений Игоря/Егора)
**Статус:** утверждён к планированию
**Предшественник:** [`2026-06-05-mpstats-tools-partner-course-design.md`](2026-06-05-mpstats-tools-partner-course-design.md) (Часть 1 — раздел `/mpstats-tools`, на проде с 2026-06-08)

## Контекст

Сейчас юзер сервиса MPSTATS, который хочет бесплатный курс по инструментам, проходит
через промежуточный лендинг `mpstats.academy/service?from=module_landing_page`:
форма (имя/телефон/email) → ручка на hostemo регистрит его в старой LMS (SkillSpace)
→ роутинг. Три прослойки. Цель — выпилить SkillSpace и максимально бесшовно приземлять
человека на **нашу платформу** в нужный урок партнёрского курса `07_instruments`.

## Критические ограничения (из встречи + переписки 2026-06-10/11)

1. **Полного SSO нет.** У MPSTATS нет OAuth-провайдера, академия во внешнем контуре.
2. **Игорь — только фронтенд. Бэкенда у него нет** (бэк на стороне Тильды, доступа нет).
   → **HMAC-подпись payload невозможна на day-1**: подписать можно только на бэке с
   секретом; во фронтовом JS секрет утечёт. Поэтому **trusted-ветка зажить не может**,
   пока не появится подписывающий бэкенд.
3. Передача — голые GET-параметры: `name, phone, email, module_code?`. Лиды Игорь
   параллельно отдаёт в AmoCRM на своей стороне (их разбирает консалтинг — важно
   качество, т.е. живые email).

## Ключевое архитектурное решение

**GET-параметры — НЕ доказательство личности.** Любой может дёрнуть
`ручка?email=victim@x.com`. Отсюда разделение по уровню доверия:

- **Trusted** (MPSTATS передаёт HMAC-подписанный payload) → создаём настоящую сессию
  сразу. **На day-1 не запитывается** (нет подписанта). Код пишем и тестируем,
  оставляем **dormant** — зажжётся без переработки, если появится бэк-подписант.
- **Untrusted** (голые GET) — **день-1 реальность**. Авто-сессию выдаём только для
  **нового email** (аккаунта ещё нет → не takeover). Для **существующего** email
  авто-логин запрещён: либо у юзера уже есть наша кука сессии (тогда сразу пускаем),
  либо magic-link на его почту (подтверждение владения).

### Модель «мягкого подтверждения почты»

Новый юзер попадает в продукт **сразу, без похода в почту**. Помечаем его
`user_metadata.partner_pending_verify = true`. Баннер «подтвердите почту» + ресенд
мягко подталкивает подтвердить (ценность — качество лидов для консалтинга), но
**ничего не блокирует**.

**Гейта оплаты НЕТ** (осознанно отвергнут): партнёрский курс бесплатный, реальная
оплата идёт по карте в CloudPayments (платит владелец карты — неподтверждённый email
не создаёт платёжного риска), а гейт по новому флагу потребовал бы бэкафилла 158
действующих плательщиков и прошивки во все auth-пути — это прод-риск в биллинге ради
несуществующей угрозы. Поэтому: **биллинг не трогаем, новых полей в схеме нет,
бэкафилла нет.** Подтверждённость держим в `user_metadata`, а не в `UserProfile`.

## Архитектура

### 1. Точка входа — публичный route handler

`GET /api/partner/mpstats/enter` — **публичный** (вне route-group `(main)`). Принимает:

| Параметр | Обязательность | Назначение |
|----------|----------------|------------|
| `email` | да | ключ идентичности и дедупа |
| `name` | опц. | префилл профиля / CQ-лид |
| `phone` | опц. | CQ-лид (НЕ гейтим) |
| `module_code` | опц. | целевой модуль → урок |
| `sig`, `exp` | опц. | HMAC-подпись (trusted, dormant) |

Поток: гейт `PARTNER_COURSES_ENABLED` → валидация → резолв `module_code → lessonId`
→ определение доверия → ветвление → 302-редирект. PII в GET **не логируем**.
Лёгкий rate-limit по IP — защита от массового создания throwaway-аккаунтов.

### 2. Ветвление

| Ветка | Условие | Действие |
|-------|---------|----------|
| **Trusted** (dormant) | валидная подпись | create-or-lookup → upsert профиль → CQ-лид → `establishSession` → 302 в урок. **Без письма.** |
| **Untrusted, новый email** | подписи нет, юзера нет в БД | `createUser(email_confirm:true, metadata:{partner_pending_verify:true})` → upsert профиль → CQ-лид → `establishSession` → 302 в урок. **Без письма, мгновенно.** |
| **Untrusted, существующий + наша кука** | подписи нет, кука сессии = этот email | 302 в урок |
| **Untrusted, существующий, куки нет** | подписи нет, юзер есть, куки нет | magic-link на их email → `/partner/check-email` |

### 3. Единый примитив сессии

`establishSession(email, target)` — паттерн Yandex-callback: Supabase Admin
`generateLink({type:'magiclink', email})` → `verifyOtp({token_hash})` → `createServerClient(...).setSession(...)` с записью cookie на `NextResponse`. Trusted и
new-email ветки гасят токен сами (мгновенная сессия); existing-no-cookie — отдаёт
ссылку по почте.

### 4. Доверие (подпись) — dormant

`sig = HMAC-SHA256("{email}|{phone}|{name}|{module_code}|{exp}", SHARED_SECRET)`,
`exp` (unix s) не истёк и ≤ now+600. Канонический формат — контракт для будущего
бэк-подписанта. На day-1 секрет не задан → trusted никогда не срабатывает, всё идёт
untrusted. Код и тесты пишем сейчас, чтобы зажглось без переработки.

> **Email normalization:** `email` нормализуется к lowercase сразу при парсе, до lookup,
> createUser, сессии и проверки подписи. Канонический контракт: **подписант должен
> подписывать уже строчный email** (`old@x.com`, не `Old@X.com`).

### 5. Резолв модуля

`module_code` → `Lesson.metadata.partnerModuleKey` → `lessonId` (логику
`partner.resolveModule` выносим в публичную server-функцию). Неизвестный код / без
урока → fallback каталог `/mpstats-tools`.

### 6. Email-доставка (magic-link + ресенд верификации)

**Технический факт:** Supabase→CQ хук (`/api/webhooks/supabase-email`) обрабатывает
только `signup/recovery/email_change`; `magiclink` падает в `default` → не доставляется.

**Решение:** добавить `case 'magiclink'` в хук → стреляем **существующее** событие
`pa_doi` с confirm-ссылкой. Переиспользуем готовое DOI-правило в CarrotQuest — **нового
CQ-правила не нужно.** Этим каналом доставляются и magic-link для existing-no-cookie,
и ресенд из баннера верификации.

### 7. Мягкая верификация (баннер)

Баннер «Подтвердите почту» показывается, когда у текущего юзера
`user_metadata.partner_pending_verify === true`. Кнопка «Отправить ссылку» →
`generateLink('magiclink')` → доставка через `pa_doi` (см. §6). На успешном
`/auth/confirm` снимаем флаг `partner_pending_verify` из `user_metadata`. Ничего не
блокирует. Welcome-письмо партнёрским авто-юзерам НЕ шлём (нет спама на чужие адреса).

### 8. Лиды

CQ-лид `pa_partner_entry` с `pa_partner_source=mpstats` (+ `pa_partner_module`) при
создании/резолве юзера. Игорь параллельно отдаёт лид в Amo. Дедуп = email.

### 9. Безопасность

- PII в GET не логируем; rate-limit по IP.
- Авто-сессия — только для **нового** email; существующий без куки → подтверждение
  владения (magic-link). Подделка GET для существующего юзера сессию не создаёт.
- HMAC с коротким `exp` (dormant). Секрет — только в env, placeholder в доках.

## Границы (вне scope)

- Полный OAuth-SSO. Гейт оплаты / платформенный флаг верификации / бэкафилл — **нет**.
- Пункт меню/баннер раздела на сайте MPSTATS — задача Влада (маркетинг).
- GET-параметр в кнопках модулей сервиса — задача Кара-команды.
- Перевод **всех** регистраций на soft-verify flow — возможная будущая фаза (Егор
  пометил «может быть»), строится поверх этой.

## Критерии готовности

- [ ] Публичная ручка `GET /api/partner/mpstats/enter` принимает параметры, отдаёт лид в CQ.
- [ ] Untrusted, новый email → авто-сессия → редирект в урок без письма; `partner_pending_verify=true`.
- [ ] Untrusted, существующий с кукой → прямой редирект в урок.
- [ ] Untrusted, существующий без куки → magic-link (через `pa_doi`) → check-email.
- [ ] Trusted-ветка (с подписью) работает в тесте; без секрета — dormant.
- [ ] `module_code` резолвится; неизвестный/без урока → каталог.
- [ ] Подделка GET без подписи **не** создаёт сессию для существующего юзера.
- [ ] Баннер «подтвердите почту» для pending-юзеров + ресенд; `/auth/confirm` снимает флаг.
- [ ] Биллинг не изменён; новых полей в схеме нет; бэкафилла нет.
- [ ] PII не в логах; секрет только в env; ручка уважает `PARTNER_COURSES_ENABLED`.
