# Дизайн: Бесшовный вход из MPSTATS на платформу (Часть 2)

**Дата:** 2026-06-10
**Статус:** утверждён к планированию
**Предшественник:** [`2026-06-05-mpstats-tools-partner-course-design.md`](2026-06-05-mpstats-tools-partner-course-design.md) (Часть 1 — раздел `/mpstats-tools`, на проде с 2026-06-08)

## Контекст

Сейчас юзер сервиса MPSTATS, который хочет бесплатный курс по инструментам,
проходит через промежуточный лендинг `mpstats.academy/service?from=module_landing_page`:
форма (имя/телефон/email) → ручка на hostemo регистрит его в старой LMS (SkillSpace)
→ роутинг. Три прослойки, SkillSpace не умеет прямой вход для уже зарегистрированных.

Цель — выпилить SkillSpace из этого пути и максимально бесшовно приземлять человека
на **нашу платформу** в нужный урок партнёрского курса `07_instruments` (Часть 1).

Встреча с Игорем Подковцевым (разработчик текущего решения) 2026-06-10:
полный OAuth-SSO сознательно **не делаем** — у MPSTATS нет OAuth-провайдера, академия
во внешнем контуре. Идём «лёгким» путём: форма остаётся на стороне MPSTATS и сабмитит
на одну нашу публичную ручку с параметрами `name, phone, email, module_code?`.

## Ключевое архитектурное решение

**GET-параметры `email/phone/name` — НЕ доказательство личности.** Любой может дёрнуть
`наша-ручка?email=victim@x.com`. Поэтому из голых GET-параметров мы не имеем права
создавать сессию (account takeover). Из этого следует развилка по уровню доверия:

- **Trusted** — MPSTATS передаёт **подписанный** payload (HMAC, общий секрет). Тогда мы
  доверяем идентичности и создаём настоящую сессию сразу → бесшовно для **всех** юзеров.
- **Untrusted** — голые GET без подписи. Авто-логин возможен только при наличии нашей
  куки сессии; иначе обязателен шаг подтверждения владения email (magic-link).

**Проектируем обе ветки в одной ручке.** Поведение определяется наличием валидной
подписи. MPSTATS включит подпись на своей стороне, когда сможет — без изменений у нас.
До этого работает untrusted-ветка.

## Архитектура

### 1. Точка входа — публичный route handler

`GET /api/partner/mpstats/enter` — **публичный** (вне route-group `(main)`, которая
требует auth). Принимает:

| Параметр | Обязательность | Назначение |
|----------|----------------|------------|
| `email` | да | ключ идентичности и дедупа |
| `name` | опц. | префилл профиля / CQ-лид |
| `phone` | опц. | CQ-лид (НЕ гейтим, в отличие от Phase 45 register) |
| `module_code` | опц. | целевой модуль → урок |
| `sig` | опц. | HMAC-подпись (trusted-ветка) |
| `exp` | опц. | срок жизни подписи (Unix ts) |

Поток: валидация входа → фиксация CQ-лида → резолв `module_code → lessonId`
→ определение уровня доверия → ветвление сессии → 302-редирект (в урок или каталог).

Гейт: ручка уважает `PARTNER_COURSES_ENABLED` (как и весь раздел) — если флаг выключен,
ручка отдаёт 404/редирект на главную.

### 2. Единый примитив сессии

Один механизм на все ветки — Supabase Admin API `generateLink({ email })` → `token_hash`.
Различается только способ «погашения» токена:

| Ветка | Условие | Действие |
|-------|---------|----------|
| **Trusted** | валидная подпись | server-side `verifyOtp(token_hash)` → ставим cookie → 302 в урок. **Без письма, мгновенно.** |
| **Untrusted, новый email** | подписи нет, юзера нет в БД | создаём юзера → шлём ссылку через CQ → клик → `/auth/confirm` → сессия → урок |
| **Untrusted, существующий + есть наша кука** | подписи нет, кука сессии валидна | сразу 302 в урок |
| **Untrusted, существующий, куки нет** | подписи нет, юзер есть, куки нет | magic-link на их email (подтверждает владение) → урок |

Примитив `generateLink` + server-side `verifyOtp` (через `@supabase/ssr` для установки
cookie) — тот же, что уже используется в проекте для resend (plan B). Trusted-ветка
просто гасит токен сами, untrusted — отдаёт юзеру по почте.

### 3. Доверие (подпись)

`sig = HMAC-SHA256("{email}|{phone}|{name}|{module_code}|{exp}", SHARED_SECRET)`.
Валидность: подпись совпадает И `exp` не истёк (`exp ≤ now + 5 мин` от момента минта).
Общий секрет — в env обеих сторон (`MPSTATS_PARTNER_SIGNING_SECRET`). Защита от replay —
короткий `exp`; опционально одноразовость токена (nonce-таблица) — YAGNI на старте.

В обеих ветках email в итоге верифицирован (trusted — верифицирован MPSTATS; untrusted —
самим magic-link/OTP), поэтому отдельный DOI-шаг не нужен.

### 4. Резолв модуля

`module_code` → `Lesson.metadata.partnerModuleKey` → `lessonId`. Логику текущего
`partner.resolveModule` (сейчас `protectedProcedure`) выносим в публичную server-функцию
(прямой запрос Prisma в route handler), т.к. на момент резолва юзер ещё не авторизован.
Неизвестный код / код без урока → fallback на каталог `/mpstats-tools`.

### 5. Лиды

CQ-лид с `source=mpstats` фиксируется при **каждом** хите ручки (до завершения auth —
чтобы лид не терялся, даже если юзер не дойдёт до конца). Игорь параллельно отдаёт лид
в AmoCRM на своей стороне (consulting-команда обслуживает эти лиды). Дедуп на нашей
стороне = email.

### 6. Email-доставка

**Технический факт (проверено):** Supabase→CQ webhook (`/api/webhooks/supabase-email`)
обрабатывает только `email_action_type` ∈ {`signup`, `recovery`, `email_change`}. Тип
`magiclink` падает в `default` → письмо **не доставляется**.

**Решение:** не полагаться на auth-email хук. В ручке генерим ссылку через
`generateLink` и шлём её **напрямую через CQ-событие** (`pa_partner_magic_link` +
automation-правило в CarrotQuest). Альтернатива — добавить `case 'magiclink'` в хук;
прямой путь предпочтительнее (полный контроль текста/ссылки, не зависит от типа письма).

### 7. Безопасность

- PII (`email/phone/name`) в GET — **не логировать** параметры запроса.
- Untrusted-ветка **никогда** не ставит сессию без подтверждения владения email.
- HMAC с коротким `exp` — защита от replay.
- `SHARED_SECRET` — только в env, никогда в код/доки (placeholder).

## Границы (вне scope)

- Полный OAuth-SSO между MPSTATS и платформой.
- Пункт меню/баннер раздела на сайте MPSTATS — задача Влада (маркетинг), отдельно.
- GET-параметр в кнопках модулей сервиса — задача Кара-команды.
- Транскрибация/ingestion уроков — отдельный пайплайн (Часть 1, RAG Phase A).

## Критерии готовности

- [ ] Публичная ручка `GET /api/partner/mpstats/enter` принимает параметры и отдаёт лид в CQ.
- [ ] Trusted-ветка (валидная подпись) → авто-сессия → редирект в урок без письма.
- [ ] Untrusted, новый email → magic-link через CQ → вход → урок.
- [ ] Untrusted, существующий с кукой → прямой редирект в урок.
- [ ] Untrusted, существующий без куки → magic-link → урок.
- [ ] `module_code` резолвится в `lessonId`; неизвестный/без урока → каталог.
- [ ] Подделка GET без валидной подписи **не** создаёт сессию для существующего юзера.
- [ ] PII не попадает в логи; секрет только в env.
- [ ] Ручка уважает `PARTNER_COURSES_ENABLED`.
