# Дизайн: Текстовые и интерактивные уроки

**Дата:** 2026-06-23
**Статус:** design approved, готов к планированию
**Фазы:** A (текстовый урок) → B (интерактив) → C (редактор состава джоб)

## Проблема

Сейчас урок на платформе — только видео (Kinescope-плеер + AI-чат по транскрипту через `content_chunk`). Методологи просят:

1. **Текстовые уроки** — собирать урок из текста с форматированием, картинками, таблицами, выносками. Нужен WYSIWYG-редактор в админке для методолога без знания кода.
2. **Интерактивные уроки** (стиль Яндекс.Практикума) — прогрессивное раскрытие: текст открывается порциями по действию ученика («читать дальше», «да/нет»), с ветвлением.

Создавать и размещать такие уроки методолог должен сам из админки.

## Референсы

| Источник | Что показывает |
|----------|----------------|
| `course.mpstats.academy/course/113526/...` (LMS методологов) | Текстовый урок: цветные H1–H3, абзацы, **жирный**/курсив, нумерованные/маркированные списки, эмодзи-буллеты, картинки с обтеканием, таблицы, callout-выноски («❗ Важно»). Внизу — блок «Ваш ответ на задание» (свой rich-редактор + аудио + файлы + «Отправить на проверку» + «История проверки»). |
| `practicum.yandex.ru/trainer/marketing-for-business-free/lesson/...` | Интерактив = блочный контент + runtime раскрытия, оформлен как диалог. Гейт «Вперёд!» раскрывает следующую порцию блоков, кликнутая кнопка становится «репликой ученика». Сверху прогресс-бар из сегментов. Квиз-блоки с дропдаунами и проверкой ответа («Узнать ответ»), фидбэк 👍👎. |

## Ключевой архитектурный инсайт

**Текстовый и интерактивный урок — одна сущность.** Урок = TipTap-документ (дерево блоков, JSON). Текстовый урок — всё раскрыто сразу, гейтов нет. Интерактивный — те же блоки + блоки-гейты, которые ставят показ на паузу. Один движок, один формат хранения, один рендерер. Фаза B аддитивно надстраивается над движком Фазы A.

## Выбор технологии: TipTap

Редактор — **TipTap** (ProseMirror под капотом, React, headless). Лицензия — **MIT, бесплатно**, self-hosted без отчислений. Платный Cloud ($49+/мес) — только для managed-документов (real-time коллаборация, комментарии, история), нам не нужен: документ храним JSON-ом в своей Postgres. 10 бывших Pro-расширений с июня 2025 открыты под MIT.

**Почему TipTap, а не BlockNote/Editor.js:** интерактивные гейты Фазы B — это произвольные React-компоненты внутри потока контента; у TipTap это нативно (node view), и один документ рендерится и в админке (editable), и ученику (read-only). BlockNote (тоже MIT) — запасной вариант, если захотим полированный Notion-редактор быстрее ценой гибкости кастомных блоков. Все три бесплатны — разницы по деньгам нет.

**Подключение:** npm-зависимости в `apps/web` (`@tiptap/react`, `@tiptap/starter-kit`, расширения таблиц/картинок). Крутится целиком на клиенте, без API-ключей и внешних сервисов.

## Модель данных (аддитивные миграции)

> ⚠️ Prod Supabase `saecuecevicwjkpmaoot` — live. Все миграции аддитивные. На VPS нет pnpm/prisma → применять через Supabase Management API (паттерн `reference_supabase_migration_via_mgmt_api.md`).

```prisma
model Lesson {
  // ...существующие поля...
  contentType   LessonContentType @default(VIDEO)  // VIDEO | TEXT | INTERACTIVE — взаимоисключающий
  body          Json?                              // TipTap-документ (дерево блоков)
  contentStatus LessonContentStatus @default(PUBLISHED) // DRAFT | PUBLISHED — флоу публикации для TEXT/INTERACTIVE
}

enum LessonContentType {
  VIDEO
  TEXT
  INTERACTIVE
}

enum LessonContentStatus {
  DRAFT
  PUBLISHED
}

model LessonProgress {
  // ...существующие поля...
  progressState Json?  // интерактив: позиция раскрытия + выборы в чекпоинтах (resume + аналитика)
}
```

- `contentType` — взаимоисключающий. VIDEO — как сейчас. TEXT/INTERACTIVE — TipTap-тело.
- `contentStatus` дефолт PUBLISHED, чтобы существующие видео-уроки не попали в DRAFT. Реально используется только для TEXT/INTERACTIVE.
- `content_chunk` — переиспользуем без изменений схемы: для текстовых уроков `source_type="academy_text"`, `timecode_start/end = 0`.

## Фаза A — Текстовый урок

### A1. Создание урока из админки (закрывает пробел)

Сегодня уроки создаются **только** seed/ingest-скриптами; в `admin.ts` нет `createLesson`. Для текст/интерактив-уроков нужно создание из интерфейса.

- Новая мутация `admin.createLesson` (adminProcedure): вход — `courseId`, `title`, `contentType` (TEXT/INTERACTIVE). Создаёт `Lesson` со `contentStatus=DRAFT`, `order` = в конец курса (учитывая `@@unique([courseId, order])`), генерирует `id` (формат, совместимый с существующим — напр. `<courseId>_text_<cuid>`).
- В `CourseManager` — кнопка «Создать урок» в курсе → диалог (заголовок + тип) → создание → открывается редактор.
- **Размещение и порядок** — переиспользуем существующее: новый урок появляется в списке курса (`getCourseLessons`), порядок двигается через `moveLessonToPosition`, переименование/скрытие — как у видео-уроков. Отдельно проектировать не нужно.

### A2. Редактор (админка, TipTap)

Набор блоков по референсу №1:
- Заголовки H1–H3, абзац, **жирный**/курсив
- Списки нумерованные/маркированные
- Картинки (загрузка в Supabase Storage — переиспользуем паттерн `requestUploadUrl` → signed URL → прямой PUT, как в `MaterialFileUpload`)
- Таблицы
- Callout-выноска (акцентный блок «❗ Важно»)
- Цитата, разделитель, ссылки

Тулбар и UI редактора собираем сами (TipTap headless). Дизайн — по `docs/design-system/` (light, продуктовая система).

### A3. Рендерер (ученик)

Тот же TipTap-документ в read-only. Урок открывается на странице урока, когда `contentType=TEXT`.

### A4. Флоу публикации и индексации

- **DRAFT**: методолог сохраняет `body` сколько угодно раз (просто пишем JSON). Урок **не виден** ученикам, **не индексируется** — ноль обращений к эмбеддингам.
- **Публикация** (кнопка «Опубликовать»): `contentStatus=PUBLISHED` (виден ученикам) + индексация текущего тела в `content_chunk`.
- **Правки после публикации**: `body` сохраняется как обычно (дёшево); в индекс прилетает только по повторному «Опубликовать изменения». **Индексация всегда триггерится публикацией, не сохранением.**
- **Одно тело** (без отдельной замороженной опубликованной копии). Полноценное версионирование контента — YAGNI, отдельная фаза если понадобится. Практический эффект: пока DRAFT — никто не видит; после первой публикации правки видны ученикам сразу, индекс освежается по кнопке.

### A5. Индексация текста в `content_chunk`

При публикации: извлечь plain-text из TipTap-JSON → чанковать → эмбеддить (`text-embedding-3-small`, 1536) → upsert в `content_chunk` с `source_type="academy_text"`, `lesson_id` урока. Idempotent: снести старые чанки этого урока → залить новые. Переиспользуем существующий пайплайн эмбеддингов (`packages/ai`). AI-чат начинает видеть текстовые уроки наравне с видео.

### A6. Завершение урока

Кнопка «Завершить урок» внизу → `LessonProgress.status=COMPLETED`. Автозачёт по доскроллу не делаем (ненадёжно на длинных уроках/мобильном).

### A7. Привязка к сущностям

Текстовый урок — обычный `Lesson` в курсе → автоматически работает в треке, поиске, и в джобах (если урок добавлен в джобу). Отдельной работы по интеграции с этими разделами не нужно.

## Фаза B — Интерактив (поверх движка A)

### B1. Новые кастомные TipTap-ноды

- **Гейт «читать дальше»** — нод-разделитель; всё после него скрыто до клика. Текст кнопки настраивается методологом.
- **Чекпоинт-развилка** — нод с N вариантами (кнопками). Под каждым вариантом методолог кладёт **свою вложенную цепочку блоков** (может содержать собственные гейты). После вложенной цепочки поток **сходится** на общую линию — к следующему блоку после чекпоинта.

### B2. Модель ветвления — A (локальные развилки со схождением)

- Развилки **локальные**, ветки **обязаны** сойтись на общую линию (нет расходящихся-до-конца финалов, нет произвольных переходов/графа).
- Гейты **вкладываются рекурсивно**: блоки внутри ветки — такие же блоки, включая «читать дальше», поэтому ветка может быть многошаговой (напр. «Космонавт» → «сколько зарабатывают» → читать дальше → «критерии приёма» → читать дальше → «кому подходит» → схождение).
- Полный граф-редактор — за скоупом (YAGNI: сложно авторить методологу, легко собрать тупик/петлю).

### B3. Runtime раскрытия (ученик)

Движок идёт по списку блоков, раскрывает до первого непройденного гейта. По действию — следующую порцию. По выбору в развилке — вложенную ветку выбранного варианта, затем возврат на общую линию. Позиция и выборы сохраняются в `LessonProgress.progressState` (resume + аналитика).

### B4. Завершение

Дошёл до конца общей линии → `status=COMPLETED`.

### B5. Аналитика

Ответы в чекпоинтах пишутся в `progressState` — методолог видит распределение выборов.

### B6. Презентация

Чат-баблы (как Практикум) vs простой inline-reveal — решается на UI-этапе (UI-SPEC), на архитектуру не влияет.

## Фаза C — Редактор состава джоб в админке (отдельная, после A/B)

Сегодня состав «решений под задачу» (`JobLesson`) задаётся **только** `seed-jobs.ts`; админ-мутаций нет. Это пробел для **всех** типов уроков, не только текстовых.

- Новый раздел админки: у джобы — добавить/убрать/переставить уроки (drag-order по `JobLesson.order`), по образцу `CourseManager`, но для джоб.
- Новые admin-мутации в `job`-роутере: `addJobLesson` / `removeJobLesson` / `reorderJobLessons`.
- **Ортогональна редактору** — не зависит от Фаз A/B, делается после, чтобы текст/интерактив-уроки можно было раскладывать по джобам без прогона скриптов.

## За скоупом (все фазы)

- **Ответ ученика на проверку** — блок «Ваш ответ на задание», аудио, файлы, «Отправить на проверку», «История проверки» (методологи хотят, но конкретных планов нет).
- **Квиз с проверкой правильности** — «Узнать ответ», подсчёт верных ответов (родственно отложенному «ответу ученика»).
- **Полный граф-ветвление** и расходящиеся-до-конца ветки.
- **Версионирование контента** — отдельная замороженная опубликованная копия vs черновик.

## Используемая существующая инфраструктура

| Что | Где | Как используем |
|-----|-----|----------------|
| `content_chunk` (vector 1536) | `schema.prisma:365`, `source_type` дефолт `academy_audio` | текст: `source_type="academy_text"`, таймкоды 0 |
| Загрузка в Supabase Storage | `MaterialFileUpload.tsx` (`requestUploadUrl` → signed URL → PUT) | картинки TipTap |
| Управление уроками в курсе | `CourseManager.tsx` + `admin.moveLessonToPosition/updateLessonTitle/toggleLessonHidden` | размещение/порядок — как есть; добавляем только `createLesson` |
| Пайплайн эмбеддингов | `packages/ai` | индексация текста |
| `LessonProgress` | `schema.prisma:200` | + `progressState` для интерактива |
| RAG-фильтр `isHidden` | retrieval (Phase 57 hidden-lesson auto-sync) | DRAFT-уроки не индексируются и не видны |
| Миграции на prod без pnpm | Supabase Management API | аддитивные миграции |

## Порядок реализации

1. **Фаза A** — текст-уроки в проде (создание из админки + редактор + рендерер + публикация/индексация + завершение + AI-чат).
2. **Фаза B** — интерактив поверх A (гейты + развилки + runtime + аналитика).
3. **Фаза C** — редактор состава джоб (ортогонально, после A/B).
