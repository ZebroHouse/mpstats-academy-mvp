# Спека: Продуктовая воронка в админке на данных Яндекс.Метрики

**Дата:** 2026-07-20
**Статус:** готово к планированию
**Автор контекста:** сессия 20.07 (отчёт по эксперименту → диагностика трафика → это)

---

## 1. Проблема

Верх воронки и поведение внутри продукта **не видны в аналитике**.

- `/admin/analytics` знает только серверные факты: регистрация → онбординг → триал → оплата.
- Трафик (визиты, уники, каналы) живёт **отдельно** в интерфейсе Яндекс.Метрики и в аналитику не попадает.
- Между «зарегистрировался» и «оплатил» — чёрный ящик: открывал ли уроки, доходил ли до тарифов.

Практическое следствие: по июльскому эксперименту мы знаем, что конверсия триал→оплата 3,8%, но **не знаем почему**. Гипотезы «не доходят до тарифов» и «доходят, но не платят» требуют разных действий, а различить их сейчас нечем.

## 2. Что строим

Новый раздел **«Продуктовая воронка»** в `/admin/analytics` из двух частей.

### 2.1 Топ-лайн трафика (Метрика)

Визиты, уникальные посетители, динамика по месяцам/дням, доля платформы в трафике домена. Цель — не ходить в интерфейс Метрики за верхом воронки.

### 2.2 Сквозная поведенческая воронка

```
Визит → Регистрация → Диагностика (старт → финиш) →
Открыл урок → Посмотрел тарифы → Триал → Оплата
```

С конверсией между шагами и фильтром по периоду (переиспользовать существующий фильтр дат аналитики).

## 3. Источники данных и авторитетность

Воронка — **джойн двух систем**. Правило: поведение — из Метрики, деньги и подписки — из БД.

| Шаг | Источник | Ключ / поле |
|---|---|---|
| Визиты, уники | Метрика Reporting API | `ym:s:visits`, `ym:s:users`, фильтр `ym:s:startURL=*'*platform.mpstats.academy*'` |
| Регистрация | Метрика-цель + сверка с БД | goal `540626668` / `UserProfile.createdAt` |
| Диагностика старт | Метрика-цель | `540626712` |
| Диагностика финиш | Метрика-цель | `540626734` |
| Открыл урок | Метрика-цель | `540626804` |
| Посмотрел тарифы | Метрика-цель | `540626808` |
| Клик по CTA | Метрика-цель | `540626878` |
| Вход | Метрика-цель | `540626673` |
| Триал | **БД** | `Subscription.status='TRIAL'` |
| **Оплата / выручка** | **БД (authoritative)** | `Payment.status='COMPLETED'` |

**Важно:** цель `platform_payment` (`540626853`) в Метрике используется только как поведенческий шаг. Деньги считаем **исключительно из БД** — клиентская цель может не сработать (блокировщики, уход со страницы).

**Исключения из БД-части:** обязательно применять текущее правило — `UserProfile.isTest = false` И `SubscriptionPlan.hidden = false` (см. `packages/api/src/utils/test-exclusion.ts`).

## 4. Ограничения Метрики (влияют на архитектуру)

1. **Лимиты API:** 200 запросов / 5 минут, 5 000 / сутки, ≤3 параллельных на токен. Превышение → HTTP 429. **Прямые запросы из UI недопустимы.**
2. **Счётчик 94592073 стоит на весь домен** `mpstats.academy`. Для метрик платформы **обязателен фильтр** по `startURL`. Без него цифра завышена примерно втрое (10 541 против 3 582 визитов за 30 дней).
3. **Атрибуция источников неполная:** тип источника определяется лишь у ~22% визитов, UTM — у ~6%. Разбивку по каналам в воронку **не тащим** (не сойдётся с итогом). Только абсолютные значения шагов.
4. **Права токена — `analyst`:** чтение отчётов и целей есть, запись (создание/правка целей) — нет (403). Для этой задачи достаточно; если понадобятся новые цели — заводит владелец счётчика (m@mpstats.io) или нужен апгрейд до `edit`.

Подробности доступа и рабочие примеры запросов: `docs/yandex-metrika-api.md`.

## 5. Архитектура

### 5.1 Кеш — крон-снапшот в БД

Прямые запросы к Метрике из процедур запрещены (лимиты + латентность). Схема:

```
Cron (раз в час) → тянет Метрику → пишет снапшот в БД
                                        ↓
                          tRPC-процедуры читают только снапшот
                                        ↓
                                    UI админки
```

**Новая таблица** (название на усмотрение реализации, напр. `MetrikaSnapshot`):

| Поле | Тип | Смысл |
|---|---|---|
| `id` | String @id | |
| `metricKey` | String | что за метрика: `visits`, `users`, `goal_540626804` и т.д. |
| `day` | Date | день, за который значение |
| `value` | Int | значение |
| `fetchedAt` | DateTime | когда снято |

Индекс `@@unique([metricKey, day])` — идемпотентный upsert, повторный прогон крона не плодит дубли.

Миграция **аддитивная** (только новая таблица), схему существующих таблиц не трогаем. Применять по правилу проекта: `packages/db/prisma/schema.prisma` — источник истины, `prisma db push` только из этого репозитория.

### 5.2 Cron-эндпоинт

По образцу существующих кронов (`apps/web/src/app/api/cron/…`), защита через `CRON_SECRET`.

- Тянет данные за последние N дней (окно перекрытия ~7 дней, чтобы догонять доуточнённые Метрикой цифры).
- Идемпотентный upsert по `(metricKey, day)`.
- Ошибка Метрики (429/5xx) не роняет крон — логируем и выходим, снапшот остаётся прежним.
- Ретрай с бэкоффом на сетевые сбои (на VPS есть гоча с IPv6 — уже решается `NODE_OPTIONS=--dns-result-order=ipv4first` в docker-compose).

### 5.3 tRPC-процедуры

В `packages/api/src/routers/admin-analytics.ts`, `adminProcedure`:

- `getTrafficOverview({ from, to })` — визиты/уники по дням из снапшота.
- `getProductFunnel({ from, to })` — шаги воронки: цели из снапшота + триал/оплаты из БД, с конверсиями между шагами.

Читают **только БД**, к Метрике не ходят.

### 5.4 UI

Новый таб/секция в `/admin/analytics` по паттерну Phase 63 / таба «Ассистент» (`v1.31`). Переиспользовать существующий фильтр периода. Чарты — Chart.js, как в остальной аналитике.

**Фичефлаг не нужен** — раздел admin-only, на клиента не раскатывается.

### Отдельное решение: стабильность токена

Текущий токен — **личный OAuth**, привязан к аккаунту, который проходил авторизацию. Если у аккаунта заберут доступ к счётчику — интеграция сломается молча (крон будет писать ошибки, снапшот застынет).

Варианты: (а) оставить как есть и добавить алерт на протухание снапшота; (б) выделенный сервисный аккаунт с гостевым доступом «только просмотр» к счётчику. Рекомендация — (б) для прода, но не блокер для первой версии.

## 7. Критерии готовности

- [ ] Крон стабильно наполняет снапшот, повторный прогон не плодит дублей.
- [ ] В `/admin/analytics` виден трафик за выбранный период без похода в Метрику.
- [ ] Воронка показывает все шаги от визита до оплаты с конверсиями.
- [ ] Цифры трафика **сходятся с интерфейсом Метрики** при том же фильтре по `platform.` (сверка вручную на одном периоде).
- [ ] Оплаты в воронке **совпадают с текущим табом выручки** (обе из БД, не должны расходиться).
- [ ] Тестовые аккаунты и скрытый тариф исключены из БД-шагов.
- [ ] Падение Метрики не роняет админку — показываем последний снапшот + отметку «данные на \<дата\>».

## 8. Что осознанно НЕ делаем

- **Разбивку воронки по каналам/UTM** — атрибуция покрывает 6–22%, цифры не сойдутся с итогом. Отдельная задача после UTM-разметки размещений.
- **Создание новых целей в Метрике** — нет прав, и текущих 8 достаточно.
- **Реалтайм** — часового снапшота хватает, аналитика не операционная.
- **Фичефлаг** — админка.
