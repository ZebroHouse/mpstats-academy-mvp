# Скидочные промокоды и амбассадор-скидки — дизайн

**Дата:** 2026-07-13
**Ветка:** `feature/discount-promo-codes` (worktree от чистого master, слияние — когда владелец закончит с остальными недоделками)
**Статус:** дизайн утверждён владельцем, готов к плану реализации

## Проблема

Сегодня коды дают только **бесплатное время**:
- `PromoCode` — создаёт comp-подписку на `durationDays` (`packages/api/src/routers/promo.ts`), денежных полей нет.
- `ReferralCode` (амбассадор, выдаётся руками) — начисляет `refereeTrialDays` триала при регистрации (`apps/web/src/lib/referral/issue.ts:164-169`).

Нужно уметь выдавать клиенту **скидку на покупку** — в процентах или фиксой в рублях. Механику расширяем и на промокоды, и на амбассадор-коды. Клиентскую p2p-рефералку не трогаем.

## Требования (согласовано с владельцем)

1. **Один клиент — одно погашение.** Повторно активировать нельзя. Реализуется per-user уникальностью.
2. **Любое число клиентов** может активировать один и тот же код (глобальный лимит опционален через `maxUses`).
3. **Тип скидки — процент ИЛИ фикса в рублях.** Глубина произвольная (0–100% / любая сумма), без искусственного потолка. Нужен **пол цены** — списание не ниже `MIN_CHARGE_RUB`.
4. **Скидка единоразовая — только на первый платёж.** Рекуррентные списания идут по полной цене.
5. **Стакинг с тарифами за длительность** (когда/если появятся) — аддитивный/последовательный: сперва цена тарифа, затем на неё промо-скидка. Многомесячных тарифов сейчас нет — механику НЕ строим, только закладываем скидку как чистую функцию от финальной базовой цены.

**Амбассадор-код** может давать дни И/ИЛИ скидку (комбинация допустима). **Промокод** — режим дни XOR скидка.

## Ключевые факты по коду (обоснование дизайна)

- **Цена считается в одном месте:** `billing.ts → initiatePayment`, `amount: plan.price`. Оно же кормит PENDING `Payment.amount` и чек 54-ФЗ (`buildReceipt`). Единственная точка вставки скидки.
- **Рекуррент задаётся на стороне CloudPayments** в момент первого платежа через виджет, параметр `recurrent.amount`. Сейчас его **не передают** → CP берёт для рекуррента сумму первого платежа (обёртка: `apps/web/src/lib/cloudpayments/widget.ts` — `amount: options.recurrent.amount ?? options.amount`). Значит для «скидки только на первый платёж» достаточно **явно передать `recurrent.amount = plan.price`**, а `amount` — со скидкой.
- **Погашение промокода уже имеет паттерн one-per-user:** `PromoActivation` c `@@unique([promoCodeId, userId])` (`packages/db/prisma/schema.prisma`).
- **Амбассадор-редемпшен уже персистит связь юзер↔код:** при регистрации создаётся строка `Referral` (`referredUserId @unique` ↔ `codeId → ReferralCode`) — `apps/web/src/lib/referral/issue.ts:148-160`. Отложенную скидку читаем через неё, **новая таблица не нужна**.

## Асимметрия применения

| Источник скидки | Когда вводится | Когда применяется |
|---|---|---|
| **Промокод** | юзером на странице оплаты (поле купона) | сразу к этому платежу |
| **Амбассадор** | по ссылке при регистрации (`?ref=CODE`) | отложенно — при первой покупке юзера |

Поэтому амбассадор-скидка «лежит» на коде и читается в момент оплаты через строку `Referral` юзера.

## Модель данных (аддитивно, без новых таблиц)

```prisma
enum DiscountType { PERCENT, FIXED }        // новый

model PromoCode {                            // +2 поля
  // ...существующие...
  discountType   DiscountType?              // задан → «скидочный» код
  discountValue  Int?                       // % (1–100) при PERCENT; рубли при FIXED
  // durationDays остаётся: код либо «на дни», либо «на скидку» (XOR)
}

model ReferralCode {                         // амбассадор, +2 поля
  // ...существующие (refereeTrialDays и пр.)...
  discountType   DiscountType?
  discountValue  Int?
  // амбассадор может дать дни И/ИЛИ скидку
}

model Referral {                             // +1 поле
  // ...существующие...
  discountConsumedAt  DateTime?             // отметка «скидка израсходована» (one-time)
}
```

## Поток применения скидки (единственная точка — `initiatePayment`)

1. **Определить применимую скидку** для юзера+плана:
   - если на странице оплаты введён скидочный промокод → валидировать (активен, не истёк, `currentUses < maxUses`, юзер ещё не гасил — по `PromoActivation`);
   - иначе → амбассадор-скидка юзера: строка `Referral` → `codeId` → `ReferralCode` с discount-полями, где `discountConsumedAt == null` и код валиден;
   - **приоритет:** введённый промокод > амбассадор, **без суммирования** (обе — скидки на первую покупку, не стакаем).
2. **Посчитать** `firstAmount = computeDiscount(plan.price, type, value)`:
   - `PERCENT`: `price − round(price × value / 100)`;
   - `FIXED`: `price − value`;
   - клампить в пол `MIN_CHARGE_RUB` (нельзя списать ≤ 0). Дефолт `MIN_CHARGE_RUB = 1`.
3. `recurrentAmount = plan.price` (полная цена).
4. PENDING `Payment.amount = firstAmount`; `buildReceipt(firstAmount)`; для промокода проставить `Subscription.promoCodeId` (поле уже есть). Вернуть `{ amount: firstAmount, recurrentAmount, ... }`.

## Рекуррент — скидка только на первый платёж

В вызовах виджета (`apps/web/src/app/pricing/page.tsx` + `(main)/billing/page.tsx` + `(admin)/admin/billing-test/page.tsx`) передавать `recurrent.amount = recurrentAmount`. Для покупок без скидки поведение не меняется (`recurrentAmount === firstAmount === plan.price`).

**Линчпин на этап реализации:** проверить, что CloudPayments честно поддерживает рекуррент-сумму, отличную от первого списания (обёртка поддерживает). План Б, если нет: первый платёж не-рекуррентным, отдельная регистрация рекуррента на полную цену.

## Учёт погашения (consume-on-success)

Код «сгорает» только при **успешной оплате** (вебхук CloudPayments `COMPLETED`), не при вводе — чтобы брошенный чекаут не жёг попытку. На успехе:
- **промокод** → создать `PromoActivation(promoCodeId, userId, subscriptionId)` + `currentUses++` (unique-констрейнт = жёсткая защита от повтора);
- **амбассадор** → если оплачено меньше полной цены и `Subscription.promoCodeId` не проставлен → выставить `Referral.discountConsumedAt = now`.

## UI

- **Страница оплаты:** поле ввода купона → валидация → показ новой цены. Для юзера с амбассадор-скидкой — **авто-применение** и плашка «скидка N% по коду AMB-… применена» (новый query `billing.getApplicableDiscount`).
- **Admin — промокоды:** переключатель режима (дни / скидка → тип %/фикса + значение).
- **Admin — амбассадор-коды:** опциональные поля скидки рядом с `refereeTrialDays`.

## Дефолтные решения

| Вопрос | Решение |
|---|---|
| Режим промокода | дни XOR скидка |
| Режим амбассадора | дни И/ИЛИ скидка |
| Два кода одновременно | введённый промокод > амбассадор, без стака |
| Пол цены | `MIN_CHARGE_RUB`, дефолт 1 ₽ |
| Погашение | consume-on-success (вебхук COMPLETED) |
| Скидка на рекуррент | нет (только первый платёж) |

## Что НЕ входит в скоуп

- Многомесячные тарифы за длительность и их стакинг со скидкой (только закладываем совместимость).
- Скидка на рекуррентные списания (технически возможна через `recurrent.amount`, но сценарий-призрак).
- Клиентская p2p-реферальная программа (`ReferralBonusPackage`) — не трогаем.
- Суммирование двух скидочных кодов.
