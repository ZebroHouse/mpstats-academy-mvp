## 1. Цель

Дать методологу авторить **интерактивные** уроки поверх движка Фазы A: с пошаговым раскрытием контента («читать дальше») и развилками-чекпоинтами (ученик выбирает вариант → видит свою ветку → возвращается на общую линию). Ученик проходит урок, его позиция и выборы сохраняются (resume + аналитика).

Тип урока `INTERACTIVE` уже существует в модели (`Lesson.contentType`), и редактор уже умеет создавать такой урок (Фаза A). Фаза B добавляет **интерактивные блоки + runtime раскрытия**.

## 2. Решения (зафиксированы на брейнсторме 2026-06-24/25)

| Вопрос | Решение |
|--------|---------|
| Презентация у ученика | **Inline-reveal** (раскрытие на месте; гейт = кнопка между блоками; варианты развилки = группа кнопок инлайн). Чат-баблы — отвергнуто (отдельный визуальный слой, больше UI-работы). На архитектуру не влияет; финально подтверждается на UI-этапе. |
| Аналитика выборов | **Только сохранять в `progressState`.** Дашборд распределения ответов — НЕ в Фазе B (отдельная задача позже). |
| Поведение чекпоинта | **Выбор обязателен, фиксируется.** Чтобы идти по общей линии дальше, ученик обязан выбрать вариант. После прохождения вложенной ветки выбор зафиксирован: на resume показывается выбранная ветка, переключать нельзя. |
| Авторинг вложенных веток | **WYSIWYG внутри ноды** (`NodeViewContent`): методолог пишет блоки варианта прямо в контейнере варианта, включая собственные гейты. |
| Вставка гейта/развилки | **Кнопки в тулбаре + слэш-команда `/`** (минимальный suggestion-плагин — сейчас slash-меню в редакторе нет). |

## 3. Модель данных

### 3.1 TipTap-ноды (общие `lessonEditorExtensions`)

Три новых кастомных нода, разделяемых редактором и рендерером (принцип Фазы A — один источник правды для схемы блоков):

- **`revealGate`** — плоский блок-разделитель.
  - Атрибуты: `{ id: string, buttonLabel: string }` (`buttonLabel` дефолт «Читать дальше»).
  - Контента не держит (`atom`-подобный блок, как горизонтальный разделитель).
  - Семантика: всё после него на **текущем уровне вложенности** (до следующего гейта/чекпоинта/конца уровня) скрыто, пока кнопка не нажата.
- **`checkpoint`** — блок-контейнер развилки.
  - Атрибуты: `{ id: string }`.
  - `content: checkpointOption+` (≥1, на практике ≥2).
- **`checkpointOption`** — один вариант развилки.
  - Атрибуты: `{ id: string, label: string }` (текст кнопки варианта).
  - `content: block+` — вложенная цепочка блоков варианта; **может содержать собственные `revealGate`** (рекурсия).

**Стабильный `id`:** генерируется (uuid) при вставке ноды, хранится в doc JSON. Нужен для двух вещей:
1. **resume** — выборы и пройденные гейты привязаны к id, переживают мелкие правки тела урока;
2. **аналитика** — `checkpointId`/`optionId` — устойчивый ключ распределения выборов.

### 3.2 Ветвление — модель A (локальные развилки со схождением)

- Развилки **локальные**, ветки **обязаны** сойтись на общую линию. Схождение **автоматическое** — по концу `content` варианта поток возвращается к блоку после `checkpoint`. Нет расходящихся-до-конца финалов, нет произвольного графа/переходов.
- Гейты вкладываются **рекурсивно**: блоки внутри варианта — такие же блоки, включая `revealGate`, поэтому ветка может быть многошаговой (пример: «Космонавт» → «сколько зарабатывают» → читать дальше → «критерии приёма» → читать дальше → «кому подходит» → схождение).
- Полный граф-редактор — за скоупом (YAGNI: трудно авторить, легко собрать тупик/петлю).

### 3.3 Миграция

Аддитивная, применяется к prod-Supabase через Management API (см. `reference_supabase_migration_via_mgmt_api`; следить за зомби idle-in-tx сессиями, блокирующими `ALTER` — `pg_terminate_backend` залежавшихся):

```sql
ALTER TABLE "LessonProgress" ADD COLUMN "progressState" JSONB;
```

(Поле было специфицировано в общей спеке, но в Фазе A НЕ добавлено.)

## 4. Редактор (авторинг)

### 4.1 Node-views (ветка `editable: true`)

- **Гейт** — чип-разделитель «📖 Читать дальше», инлайн-правка `buttonLabel`, кнопка удалить.
- **Чекпоинт** — карточка-контейнер:
  - секции/вкладки вариантов, каждая = `NodeViewContent` (`block+`), где методолог печатает блоки варианта;
  - править `label` варианта; добавить / удалить / переставить вариант (минимум через простые контролы — по образцу контекст-тулбара таблиц Фазы A).

### 4.2 Вставка

- Кнопки в существующем тулбаре редактора («Читать дальше» / «Развилка»), по образцу текущих контролов таблиц/картинок.
- **Слэш-меню `/`**: минимальный suggestion-плагин (`@tiptap/suggestion`) с двумя пунктами на старте (гейт, развилка). Это новый механизм в редакторе.

### 4.3 Ограничения авторинга

- `checkpointOption.content` не пуст (схема `block+`).
- Допустимо вкладывать гейт в вариант. **Вкладывать чекпоинт в чекпоинт — запрещаем** на уровне схемы (`checkpointOption` content исключает `checkpoint`), чтобы держать модель A простой и избежать графа. *(Гейт внутри варианта разрешён; чекпоинт внутри варианта — нет.)*

## 5. Рендерер ученика

### 5.1 Маршрутизация

Страница урока ветвится по `contentType`:
- `VIDEO` → как сейчас (Kinescope-плеер).
- `TEXT` → существующий read-only `LessonBodyRenderer` (Фаза A) без изменений.
- `INTERACTIVE` → новый `InteractiveLessonRenderer`.

### 5.2 Алгоритм (рекурсивный walker)

Walker идёт по плоскому массиву блоков (`doc.content` или `checkpointOption.content` — один и тот же код рекурсивно):

1. **Обычные блоки** (paragraph/heading/image/table/list/…) копятся в «сегмент». Сегмент рендерится через read-only `<EditorContent>` со срезанным суб-doc `{ type: 'doc', content: <сегмент> }` — **тем же рендерером и `lessonEditorExtensions`, что и Фаза A**. Это даёт:
   - визуальный паритет с Фазой A бесплатно (та же `.lesson-content` типографика, те же node-views картинок/таблиц);
   - обход проблемы «node-view не может скрыть соседние блоки» — сегментацией владеет walker, а не нода.
2. **`revealGate`**: завершает текущий сегмент. Если гейт **не пройден** (`id ∉ revealedGateIds`) — рендерит кнопку `buttonLabel`, всё ниже на этом уровне скрыто, walker останавливается. Если **пройден** — рендерит следующий сегмент (рекурсия дальше по списку). Клик по кнопке: добавляет `id` в `revealedGateIds`, сохраняет `progressState`.
3. **`checkpoint`**: завершает текущий сегмент. Рендерит кнопки вариантов (`label`).
   - Если **не выбран** (`id ∉ checkpointChoices`) — выбор обязателен, общая линия ниже скрыта, walker останавливается на чекпоинте.
   - Если **выбран** — рекурсивно прогоняет walker по `content` выбранного варианта (его гейты обрабатываются так же), затем продолжает общую линию (блоки после чекпоинта). Кнопки вариантов в зафиксированном состоянии (выбранный подсвечен, остальные неактивны).
   - Клик по варианту: пишет `checkpointChoices[checkpointId] = optionId`, сохраняет `progressState`.

### 5.3 Состояние

- Reveal-состояние — локальный React-стейт рендерера, инициализируется из `progressState` (resume).
- На каждое действие (раскрытие гейта / выбор варианта) — оптимистичный апдейт стейта + мутация сохранения `progressState`.

## 6. Прогресс / resume / завершение

### 6.1 Форма `progressState`

```jsonc
{
  "version": 1,
  "revealedGateIds": ["<gateId>", ...],
  "checkpointChoices": { "<checkpointId>": "<optionId>", ... }
}
```

Позиция раскрытия **выводится** из этих данных (walker по тем же правилам). Отдельно хранить индекс не нужно.

### 6.2 Мутация сохранения

Новая мутация в `learning`-роутере (`protectedProcedure`), напр. `saveInteractiveProgress({ lessonId, progressState })`:
- Валидация `progressState` через zod (version, массив строк, map строк).
- Idempotent upsert на `LessonProgress` (ключ — пользователь+урок, как у существующего прогресса).
- НЕ ставит `COMPLETED` сама — завершение отдельным действием (ниже).

### 6.3 Завершение

- Урок завершён, когда ученик дошёл до конца **общей линии**: нет непройденных гейтов на верхнем уровне и все встреченные на пути чекпоинты отвечены.
- Завершение триггерит существующую `learning.completeLesson` (как TEXT-урок в Фазе A — кнопка «Завершить урок» появляется/активируется, когда walker дошёл до конца). Завершение и прогресс — раздельные мутации (как в Фазе A: прогресс ≠ completion).

## 7. Аналитика

Только сохранение: `checkpointChoices` лежит в `progressState`, методолог сможет посчитать распределение позже. **UI/дашборд распределения — НЕ в Фазе B** (отдельная задача).

## 8. Индексация контента (AI-чат / поиск)

`publishLesson` уже индексирует тело урока в `content_chunk` (`source_type='academy_text'`). `extractPlainText` в `packages/ai/src/text-index.ts` должен **рекурсивно обходить новые ноды**:
- `revealGate` — текста нет (только `buttonLabel`; в индекс не нужен или опционально);
- `checkpoint` → `checkpointOption[]` → их `content` (рекурсивно) — текст всех веток индексируется, чтобы AI-чат и `searchLessons` «видели» контент интерактивного урока целиком, включая ветки.

Добавить обработку этих block-типов в обход `extractPlainText` (сейчас он знает только Фаза-A-блоки).

## 11. Гочи из Фазы A (не переучивать заново)

- **TipTap v3.** `immediatelyRender: false` на каждом `useEditor` (Next SSR). Link — в StarterKit. Table-расширения — именованные экспорты.
- **`useEditor` не ре-рендерит на изменение выделения** — форс-рендер на `selectionUpdate`/`transaction` (уже сделано в `LessonEditor`; reveal-стейт рендерера — собственный React-стейт, ок).
- **Типографика:** scoped `.lesson-content` (НЕ `@tailwindcss/typography`) — расширить для новых блоков.
- **Миграции на prod:** только аддитивные, через Mgmt API; зомби idle-in-tx сессии могут блокировать `ALTER` (`pg_terminate_backend` залежавшихся).
- **`publishLesson` читает тело из БД** → редактор обязан `updateLessonBody` (сохранить) ДО `publishLesson`, иначе индексируется протухшее тело.
- **`overflow-x: clip` (не `hidden`)** на html/body — иначе ломается `position: sticky` во всём приложении (уже починено в Фазе A).
