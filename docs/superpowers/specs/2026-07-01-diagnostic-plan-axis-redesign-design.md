**Триггер:** фидбэк с реальных аккаунтов (Марина Кисельникова, Татьяна Непа) — план непрозрачен, «стена уроков», сломанный «разбор ошибок».

---

## 1. Контекст и проблема (факт-чек по коду + прод-данным)

Разобрали текущий флоу диагностики тремя исследователями + проверкой прод-данных. Выводы:

1. **Рекомендация задач работает и НЕ замокана.** `getRecommendedJobsFromGaps` (`packages/api/src/utils/job-matcher.ts:54-126`) — реальный запрос: 2 слабейшие оси → published `Job` с пересечением по `axes` + маркетплейс-фильтр → скор → top-3. Но в UI карточка задачи **не объясняет «почему именно она»** (нет привязки к оси/баллу) → у пользователя ощущение «набор задач не связан с результатом».

2. **«Разбор ошибок» мёртв для всех новых юзеров.** Секция `errors` строится из `sourceData.lessonIds` неверных ответов (`diagnostic.ts:264-292`). Статик-дек методологов (Phase 59, 30 вопросов, `static-deck.ts:16-24`) **не несёт** `sourceLessonIds`/`sourceChunkIds` → `submitAnswer` пишет `sourceData: undefined` (`diagnostic.ts:696-700`) → фильтр ошибок всегда пуст → секция вырезается. Подтверждено на данных: у Татьяны (диагностика 01.06, старый LLM-банк с RAG-привязкой) errors=8 уроков; у Марины (01.07, статик-дек) errors отсутствует. Это **системно**, а не про аккаунт.

3. **План раздувается и наполняется полу-случайно.** Лимиты секций: `deepening ≤30 + growth ≤20 + advanced ≤15` = потолок 65 + безлимитные errors/custom → возможен план на 80 уроков. `growth`/`advanced` берут **первые по `order`**, без сортировки по релевантности (`diagnostic.ts:356-382`) → в «Продвинутый уровень» попадают нерелевантные уроки (наблюдалось «как открыть VPN» в advanced).

4. **Две несовместимые таксономии.** Радар показывает 5 **осей-компетенций**. План раскладывает по абстрактным **уровням** (Углубление/Развитие/Продвинутый), которые по сути = балл по оси в переодетом виде (`<67 → deepening`, `67-99 → growth`, `100 → advanced`). Пользователь вынужден мысленно переводить одно в другое.

5. **Экран результатов — «стена из трёх списков»:** CTA «готов план» → блок «Рекомендованные задачи» → блок «Рекомендованные уроки», каждый ведёт в своё место, без объяснения связи (`apps/web/src/app/(main)/diagnostic/results/page.tsx`). Плюс «Начать обучение» у юзера без плана ведёт не в план, а в `/learn/library` — то самое «полотно».

## 2. Цели и не-цели

**Цели:**
- Единая таксономия — **ось-компетенция** — на всех экранах (радар = результаты = план). Ноль ментального перевода.
- Экран результатов: понятный один маршрут + объяснение сущностей + объяснение «почему эта задача».
- Персональный план: по осям, слабые сверху, ограниченный по размеру, релевантный подбор уроков.
- Оживить «разбор ошибок» — привязать 30 вопросов к урокам (RAG-автоматом, беглая ручная проверка), флаг на уроке в своей оси.

**Не-цели:**
- Не возвращаем LLM-генерацию вопросов. Статик-дек остаётся источником.
- Не трогаем `LessonProgress` (прогресс/завершение уроков) — жёсткое правило проекта. План можно регенерить сколько угодно, прогресс живёт отдельно по `lessonId`.
- Не меняем скоринг диагностики (`calculateSkillProfileFromAnswers`) и подбор задач (`job-matcher.ts`) по существу — только добавляем в UI объяснение «почему».
- Не трогаем биллинг/гейтинг оплаты.

## 3. Единый хребет: ось-центричная модель

Спина всего флоу — **5 канонических осей** (`diagnostic.ts:81-87`):
`ANALYTICS` (Аналитика) · `MARKETING` (Маркетинг) · `CONTENT` (Контент) · `OPERATIONS` (Операции) · `FINANCE` (Финансы).

У пользователя после диагностики есть `SkillProfile` — балл 0–100 по каждой оси. Всё строится вокруг этих 5 осей, отсортированных **по возрастанию балла** (слабые сверху):

- **Радар** — 5 осей, 2 слабейшие подсвечены.
- **Экран результатов** — топ-2 слабые оси (задачи + короткий пощупать-список уроков).
- **Персональный план** — все 5 осей блоками; слабые развёрнуты сверху, сильные свёрнуты снизу.

Абстрактные «уровни» (Углубление/Развитие/Продвинутый) как **верхняя таксономия удаляются**. Подбор сложности не исчезает — становится логикой выбора уроков **внутри** оси (см. §5). «Разбор ошибок» — не отдельная секция, а **флаг** на конкретных уроках внутри их оси (см. §7).

## 4. Персональный план — переработка

### 4.1 Модель данных

Новый формат `AxisLearningPath` (`version: 3`) в `packages/shared/src/types/index.ts` рядом с существующим `SectionedLearningPath` (v2):

```ts
interface AxisLearningPathSection {
  axis: SkillCategory;          // 'ANALYTICS' | 'MARKETING' | ...
  label: string;                // «Аналитика»
  score: number;                // 0-100 из SkillProfile
  tier: 'weak' | 'medium' | 'strong';  // из score, для стиля/сворачивания
  collapsed: boolean;           // strong && нет ошибок → true
  jobIds: string[];             // рекомендованные задачи этой оси (0-1)
  lessonIds: string[];          // подобранные уроки оси (капнутые, §4.3)
  errorLessonIds: string[];     // уроки-разборы ошибок этой оси (флаг, §7)
}
interface AxisLearningPath {
  version: 3;
  sections: AxisLearningPathSection[];   // отсортированы по score asc
  generatedFromSessionId: string;
  previousSkillProfileId?: string;
}
```

`parseLearningPath` (`index.ts:297-303`) расширяется: распознаёт `version:3` → `AxisLearningPath`; v2/flat остаются для обратной совместимости (§8).

### 4.2 Порог tier (только для подачи, не отдельная таксономия)
- `weak`: score < 50 (0 или 1 из 3 верных)
- `medium`: 50 ≤ score < 100 (2 из 3)
- `strong`: score = 100 (3 из 3)

(Пороги — константы, тюнятся в плане. `strong` жёстко = 100, т.к. с 3 вопросами на ось это единственный «идеал».)

### 4.3 Сборка секций — новая `generateAxisPath(prisma, skillProfile, sessionAnswers)`

Заменяет `generateSectionedPath` (`diagnostic.ts:254-418`). Алгоритм:

1. Для каждой из 5 осей — вычислить `score`, `tier`, подобрать задачу (первая из `getRecommendedJobsFromGaps` для этой оси, если есть) и уроки (§5).
2. Собрать `errorLessonIds` этой оси (§7).
3. `collapsed = tier === 'strong' && errorLessonIds.length === 0`. Ось с ошибками **никогда** не сворачивается.
4. Отсортировать секции по `score asc`. Пустые (нет ни задач, ни уроков, ни ошибок) — отбросить.
5. Применить капы (§4.4).

### 4.4 Размер плана (решение владельца: лимит на ось + общий потолок)
- `PER_AXIS_LESSON_CAP = 5` — не более 5 активных уроков на ось.
- `PLAN_ACTIVE_LESSON_CAP = 20` — суммарный потолок активных (не-collapsed) уроков. При превышении — резать с **сильного конца** (наименее срочные оси), но **никогда не резать `errorLessonIds`** и первую (слабейшую) ось.
- Collapsed-оси (сильные без ошибок) в общий потолок не считаются — их уроки (HARD «докачать») подгружаются лениво при раскрытии.

### 4.5 Чтение плана — `getRecommendedPath` (`learning.ts:293-473`)
Расширяется на `version:3`: отдаёт секции как есть + резолвит `jobIds`/`lessonIds`/`errorLessonIds` в полные объекты уроков/задач (реюз текущей логики обогащения). Клиент получает готовые ось-секции.

## 5. Подбор уроков внутри оси (чинит «VPN в продвинутом»)

Заменяет «первые по `order`». Для оси с баллом `score` и списком её уроков (`skillCategory`/`skillCategories` содержит ось, `isHidden=false`, видимый курс):

1. **Целевой уровень сложности от балла:**
   - `weak` → предпочесть `EASY`, затем `MEDIUM` (фундамент).
   - `medium` → `MEDIUM`, затем `HARD`.
   - `strong` → только `HARD` («докачать»).
2. **Ранжирование внутри уровня — по релевантности, не по `order`.** Основной сигнал — семантическая близость урока к оси. Реюз эмбеддингов: ранжируем уроки оси по близости к якорю оси (эмбеддинг названия оси/её топиков) через тот же `content_chunk`/`embedQuery`-механизм, что уже есть в `intent`/`searchLessons`. `order` — только тай-брейк.
3. Обрезать до `PER_AXIS_LESSON_CAP`.

Дедупликация между осями через `usedLessonIds: Set` (как сейчас), приоритет — слабейшая ось первой.

> Примечание: «VPN»-урок больше не может попасть в «Продвинутый», потому что (а) абстрактного ведра «Продвинутый» нет, урок всегда под своей осью; (б) в сильной оси берутся только `HARD`-уроки, ранжированные по релевантности — служебный EASY-урок туда не пройдёт.

## 6. Экран результатов — редизайн (`/diagnostic/results`)

Один вертикальный поток, «стена трёх списков» убирается. Сверху вниз:

1. **Хедер** — «Диагностика завершена» + общий результат.
2. **Радар навыков** — 2 слабейшие оси подсвечены + подпись «Ваши слабые зоны: X, Y».
3. **Блок «Как устроено обучение»** (новый, inline на экране — решение владельца, не тултип): короткое объяснение сущностей.
   - **Задача** — готовый маршрут из уроков под конкретную цель.
   - **Урок** — один материал: видео или текст.
4. **«С чего начать»** — задачи под слабые оси (реюз `results.recommendedJobs`). Задача №1 развёрнута с подписью **«Закрывает: {ось} — {score}% (ваша слабейшая зона)»**; остальные свёрнуты, каждая подписана своей осью. Это и есть объяснение «почему эта задача».
5. **«Или начните с отдельного урока»** — короткий пощупать-список, сгруппированный по 1–2 слабым осям, **жёсткий лимит `RESULTS_LESSON_TEASER_CAP = 5`** (не полотно). Каждый урок кликабелен.
6. **Главный CTA** — «Открыть персональный план →».

Плюс: **«Начать обучение» у юзера с планом ведёт в `/learn/plan`** (а не в library). Проверить редирект `apps/web/src/app/(main)/learn/page.tsx:25-33` — если план непустой, вести в план.

UI компоненты: правки в `results/page.tsx` + `RecommendedJobsBlock.tsx` (добавить axis-подпись из данных бэкенда) + новый компонент teaser-списка уроков по осям.

### 6.1 Экран плана (`/learn/plan`)
Секции-аккордеон по осям вместо уровней (`apps/web/src/app/(main)/learn/plan/page.tsx`): заголовок секции = «{ось} — {score}%» + tier-бейдж (🔴 слабая / 🟡 средняя / 🟢 сильная), цветной левый бордюр. Слабые развёрнуты, сильные свёрнуты. Уроки-ошибки — с флагом «⚠ Разбор ошибки» наверху своей оси. Блок задач оси — внутри секции оси. `DIAGNOSTIC_SECTION_IDS` (`plan/page.tsx:57`) заменить на список осей.

## 7. Оживление «разбора ошибок» — привязка вопросов к урокам

Решение владельца: RAG предлагает привязку, мы бегло проверяем глазами, **без передачи методологам** → не блокирует релиз.

### 7.1 Данные
Добавить в `StaticQuestion` (`static-deck.ts:16-24`) опциональные:
```ts
sourceLessonIds?: string[];       // 1-3 урока-ответки
sourceTimecodes?: Array<{ lessonId: string; label: string; start: number; end: number }>; // опционально
```

### 7.2 Скрипт авто-маппинга (одноразовый)
`scripts/diagnostic/map-questions-to-lessons.ts`: для каждого из 30 вопросов — `embedQuery(prompt + правильный вариант + explanation)` → vector-поиск по `content_chunk` (порог close) → top-1..2 урока → вывести предложение `{questionId, suggestedLessonIds, titles, score}` в JSON. Мы бегло сверяем, вписываем итоговые `sourceLessonIds` прямо в данные `static-deck.ts`. Запуск: `NODE_OPTIONS=--dns-result-order=ipv4first ... --conditions=react-server` (паттерн проекта).

### 7.3 Проброс в рантайм
- `startSession` (`diagnostic.ts:508-523`) — маппит `question.sourceLessonIds`/`sourceTimecodes` → `DiagnosticQuestion` (сейчас дропает).
- `submitAnswer` (`diagnostic.ts:696-700`) — писать `sourceData` при наличии `sourceLessonIds` (сейчас условие только на `sourceChunkIds`). `sourceData = { lessonIds, timecodes }`.
- `generateAxisPath` (§4.3) — из неверных ответов с `sourceData.lessonIds` собрать `errorLessonIds`, разложить по осям (урок → его `skillCategory`), выставить флаг + защитить от сворачивания/обрезки.

Going-forward: существующие планы не трогаются; новые диагностики получают рабочий флаг ошибок.

## 8. Обратная совместимость и миграция

- **Существующие v2/flat планы** (напр. Татьяна): `parseLearningPath` продолжает их распознавать; но UI теперь ось-центричный.
- **Миграция-на-чтение:** `getRecommendedPath` при формате ≠ v3, но при наличии `SkillProfile` у юзера — на лету пересобирает в v3 через `generateAxisPath(skillProfile)` (данные для этого есть: баллы + категории уроков; повторный проход диагностики не нужен). Опционально персистит v3 обратно.
- `rebuildLegacyLearningPath` (`utils/legacy-path-rebuild.ts`) — обновить, чтобы выдавал v3.
- `LessonProgress` не затрагивается — завершённость уроков сохраняется при любой пересборке.

## 9. Изоляция и юниты (тестируемость)

Чистые функции (юнит-тесты изолированно, паттерн проекта — утилы в `packages/api/src/utils/`):
- `generateAxisPath(skillProfile, lessons, jobs, answers)` → секции. Тесты: сортировка по score, tier-пороги, сворачивание сильных, ось-с-ошибкой не сворачивается.
- `selectAxisLessons(axis, score, lessons)` → капнутый ранжированный список. Тесты: уровень от балла, релевантность > order, cap.
- `applyPlanCaps(sections)` → капы. Тесты: per-axis 5, global 20, errors/слабейшая ось не режутся.
- `collectErrorLessonsByAxis(answers, lessons)` → `Map<axis, lessonIds>`. Тесты: пусто без sourceData; корректная раскладка по осям.
- UI-тесты `results/page` + `plan/page`: рендер ось-секций, teaser-cap, axis-подпись задач, «Начать» ведёт в план.

## 10. Роллаут и риск

- **Аддитивно, going-forward.** Новый формат плана v3 + новые поля вопросов — старые данные читаются.
- Деплой по проектному раннбуку (staging `--no-cache web` + content-check → master `--no-ff` → прод build+recreate + smoke).
- **Откат:** `git revert` фичи-мержа + редеплой. Формат v3 обратно-совместимо парсится (v2/flat не ломаются); если откат — новые v3-планы просто не читаются старым кодом → защитить fallback в `parseLearningPath` (возврат `[]` вместо throw, как сейчас).
- Без миграций схемы БД (LearningPath.lessons — Json; поля вопросов — в коде static-deck). RAG-маппинг — правка данных в коде, не в БД.
