## 1. Контекст и цель

v1.0 сквозного AI-ассистента реализован и висит на staging (`ASSISTANT_ENABLED=true`), командный UAT. Owner: **на прод катим пакетом v1.0 + концьерж** — концьерж является прод-гейтом.

**Концьерж** = административный слой поверх материального ассистента: агент знает не только материалы курсов, но и **саму платформу** — разделы, как что сделать, реф-программу пошагово, флоу диагностики, что умеет сам агент, сколько уроков в курсах. Становится «консьержем» платформы. Отвечает **строго по курируемой карте**, ничего не выдумывает.

Заодно закладываем **общий хребет** — категорию каждой реплики, которая нужна и концьержу (гейт), и будущим слоям (саппорт, аналитика).

**Вне скоупа этой сессии** (последующие слои): саппорт (детект негатива → предложить тикет → агент сам шлёт письмо), аналитика использования, 👍/👎-фидбэк.

## 2. Ключевые решения (зафиксированы в брейншторме)

| # | Решение | Выбор |
|---|---------|-------|
| 1 | Источник знаний о платформе | **Курируемый конфиг-файл в репо** (источник правды), матч через эмбеддинги. Не свободный RAG. |
| 2 | Grounding концьержа | **Строгий, но с адаптацией.** LLM формулирует ответ под язык/вопрос юзера, но **строго граундится на тексте найденных записей карты** — не добавляет шагов/кнопок/разделов/ссылок, которых в записях нет. Deep-links — только из карты (whitelist). Нет матча → честный промах. |
| 3 | Хребет | Гейт возвращает `category` (`material \| platform_help \| complaint \| off_domain`) тем же LLM-вызовом; `inDomain` производный. Одна аддитивная колонка `AssistantMessage.category`. 👍/👎 — вне скоупа. |
| 4 | Ветвление | **Одна категория → одна ветка** (KISS). Смешанный запрос падает в доминирующую категорию. |
| 5 | `complaint` в v1 | **Только детекция + запись.** Поведенчески отвечаем как `material`. Offer-тикет + агент-шлёт-письмо = слой 2. |
| 6 | Промах концьержа | Честный текст + `/support` deep-link карточка. **Только** в `platform_help`-ветке; на промахе по материалу ссылку на саппорт НЕ даём. |
| 7 | Хранение векторов карты | **В репо** (пересчитанный JSON рядом с конфигом, cosine в JS). Ноль касания прод-БД. |
| 8 | FAQ `/support` | **Сливаем в одну карту** с концьержем — единый источник правды how-to, страница FAQ читает ту же карту. |
| 9 | Факты про каталог курсов («сколько уроков / темы») | **Живой резолвер из БД** (не статический снапшот) — не устаревает при добавлении курсов. |

## 3. Хребет: категория реплики

### Гейт (`packages/ai/src/assistant/gate.ts`)
Тот же единственный LLM-вызов, что уже идёт перед ретривом, но возвращает `{ category }` вместо `{ inDomain }`:

```ts
export type ReplyCategory = 'material' | 'platform_help' | 'complaint' | 'off_domain';
export interface DomainVerdict { category: ReplyCategory; }
```

- `inDomain` становится производным: `category !== 'off_domain'`. **Квота-логика (`assistant-quota.ts`) не трогается** — считает по `inDomain`, который теперь выводится из категории.
- **Fail-open сохраняется:** ошибка/невалидный JSON → `category = 'material'` (безопаснее всего пустить как обычный вопрос, не отказать реальному селлеру).
- Системный промпт гейта расширяется определением 4 категорий. Ключевое отличие от текущего: «пользование этой платформой» (где что нажать, как отменить, реф-программа, диагностика) = `platform_help`, а **не** отказ.

### Хранение
- Аддитивная колонка `AssistantMessage.category String?` (nullable). **Единственная миграция сессии**, через Supabase Mgmt API (паттерн `reference_supabase_migration_via_mgmt_api.md`).
- Пишется на ту же реплику, где сейчас пишется `inDomain` (см. `assistant.ts` роутер). `inDomain`-колонка остаётся (обратная совместимость + квота).

## 4. Источник знаний: карта платформы

Карта = курируемый конфиг в репо (`packages/ai/src/assistant/concierge/platform-map.ts`), **единственный источник правды**. Записи двух видов.

### Амбиция и метод построения (не срезаем углы)
Цель — **полноценный помощник**, а не полусырая интерактивная документация. Карта покрывает **всю юзерскую продуктовую поверхность**: разделы, экраны, ключевые кнопки/действия, флоу — включая те, про которые owner не вспомнил. Полнота **вытаскивается из реального кода, не выдумывается**:
- **Аудит продуктовой поверхности** — параллельные субагенты (оркестратор-паттерн) проходят по юзерским роут-группам (`(main)`, `(auth)`; `(admin)` — вне скоупа), крупным компонентам, действиям юзера → выдают **структурный инвентарь**: каждый раздел / экран / кнопка / флоу с реальными `href`.
- Из инвентаря авторим записи карты; **вычитывают owner/методологи** (client-facing точность).
- Инвентарь-аудит **повторяем** — прогоняется при крупных редизайнах для сверки карты с кодом.

Ожидаемый объём — существенно больше «FAQ+навигация» (десятки записей). Матчер (§ниже) держит top-K в памяти — рост карты архитектуру не меняет.

### A. Статические записи (авторский how-to)
```ts
interface StaticMapEntry {
  id: string;                    // стабильный слаг, напр. 'cancel-subscription'
  kind: 'static';
  triggers: string[];            // перефразировки для матча (эмбеддятся)
  answer: string;                // КАНОНИЧЕСКИЙ текст ответа (markdown, без выдумки)
  deepLink?: { label: string; href: string };  // propose→click, опц.
  section: string;               // для аналитики: 'billing' | 'referral' | 'diagnostic' | ...
}
```

Обязательные тематические кластеры (не исчерпывающий список — конкретные записи даёт аудит; черновик готовит разработчик из кода + памяти, **вычитывают owner/методологи**):
- **Само-описание агента** — что концьерж умеет (отвечать про бизнес на МП, подбирать уроки/задачи, добавлять в избранное, ориентировать по платформе).
- **Реф-программа пошагово (юзер-флоу)** — как пригласить друга, что друг получает, что и когда получает реферер, как активировать. Пишется по **реальной механике**: друг — 7 дней сразу, реферер +14 **после оплаты** друга; реф-код генерится лениво в `referral.getMyState` (источники: `project_referral_pay_gated_schedule.md`, `project_referral_code_runtime_fix.md`).
- **Флоу диагностики** — как проходит, как формируются рекомендованные **уроки и задачи** (ось-центричный `AxisLearningPath v3`, top-3 задачи, разбор ошибок). Источник: `project_diagnostic_axis_redesign.md`.
- **Навигация/разделы** — `/dashboard`, `/billing`, `/profile`, `/profile/referral`, `/learn` (+ plan/favorites/library), `/diagnostic`, `/mpstats-tools`, `/support`.
- **Оставшийся FAQ** — подписка (оформить/тарифы), видео не играет, не пришло письмо-подтверждение (переносится из `FAQ_ITEMS` на `/support`).

### B. Динамические записи (живой резолвер из БД)
Для вопросов про **каталог курсов** («сколько уроков в курсе X», «какие темы») ответ строится из живой БД, не из замороженного текста:
```ts
interface DynamicMapEntry {
  id: string;                    // напр. 'course-catalog'
  kind: 'dynamic';
  triggers: string[];            // «сколько уроков в курсе», «какие темы в аналитике», ...
  resolver: 'courseFacts';       // ключ резолвера
  section: 'catalog';
}
```
Резолвер `courseFacts` (`concierge/course-facts.ts`): по распознанному курсу (или списку курсов) возвращает `title + количество опубликованных уроков + верхнеуровневые темы + deep-link на курс`. Ответ держим кратким — курс + N уроков + темы, без простыни всех названий. Уважает `isHidden=false` и `course.isHidden=false` (как остальной ретрив). Строгий grounding: источник — БД, LLM только оформляет предоставленные факты.

### Матч запроса → запись (эмбеддинги в репо)
- Скрипт `concierge/embed-map.ts` эмбеддит `triggers`(+`answer` для static) каждой записи (через существующий `embedQuery`) → пишет `platform-map.embeddings.json` (committed в репо).
- Рантайм-матчер `concierge/concierge-match.ts`: эмбеддит запрос → cosine по всем записям в памяти → **top-K записей выше порога** (K≈3-5). Широкий вопрос («из чего состоит витрина?») закрывается несколькими записями; узкий («как отменить подписку») — одной. Нет записей выше порога → честный промах. Порог и K калибруются (см. §7).
- Ноль прод-БД, ноль pgvector для этой фичи. Правка карты → прогон `embed-map.ts` → коммит JSON.

### Синхронизация FAQ `/support` ↔ карта
Страница `/support` начинает рендерить FAQ из подмножества карты (записи с флагом `showInFaq: true` или секцией), а не из локального захардкоженного `FAQ_ITEMS`. Один источник → нет дрейфа.

## 5. Пайплайн: ветвление по категории

`packages/ai/src/assistant/pipeline.ts` роутит по `category`:

| Категория | Поведение |
|-----------|-----------|
| `off_domain` | Текущий отказ (`OFF_DOMAIN_REPLY`). Карточек нет. |
| `material` | Текущий путь без изменений: `retrieveForAssistant` → `synthesizeAssistantResponse` + whitelist карточек уроков/задач. |
| `platform_help` | **Концьерж-ветка** (новая): матч по карте → static (курируемый текст + deep-link) или dynamic (резолвер) → ответ. Промах → честный текст + `/support` deep-link. |
| `complaint` | v1: детекция записана в `category`; **поведенчески = `material`** (пытаемся помочь). Спец-действий нет. |

### Концьерж-ветка (`concierge/concierge-pipeline.ts`)
1. **Матч по карте → top-K записей** выше порога (K≈3-5). Для dynamic-хита среди них — вызвать резолвер (§4.B) и положить факты из БД в контекст LLM.
2. **Есть хиты:** LLM синтезирует ответ, **адаптированный под язык и формулировку вопроса**, но **строго граундясь на тексте найденных записей** (+ факты резолвера для dynamic) — комбинирует несколько записей, ничего не добавляя сверх источника. `navLinks` = deep-links найденных записей (whitelist, обычно 1-3 самых релевантных).
3. **Промах** (нет записей выше порога): `answer = «Точно подсказать по этому не берусь, чтобы не запутать. Если что — напиши в поддержку, там помогут.»`, `navLinks = [{ label: 'Написать в поддержку', href: '/support' }]`.

**Строгий grounding — инварианты:**
- LLM **не генерирует** `href` и **не придумывает** шаги/кнопки/разделы, которых нет в найденных записях. Разрешено: переформулировать, сжать, соединить несколько записей, адаптировать тон под вопрос.
- `navLinks[].href` — только из карты (белый список, как `lessonIds` сейчас).
- Нет записей выше порога → не гадаем, честный промах.
- Температура низкая; eval-набор проверяет отсутствие выдуманных действий в ответе (§7).

## 6. Deep-link карточка + фронт + контракт данных

### Контракт (`packages/ai/src/assistant/types.ts`)
`AssistantTurnResult` расширяется:
```ts
export interface AssistantTurnResult {
  category: ReplyCategory;              // NEW (было inDomain; inDomain выводим при персисте)
  answer: string;
  lessons: AssistantLessonRef[];
  jobs: AssistantJobRef[];
  navLinks: AssistantNavLink[];         // NEW: навигационные deep-links концьержа
}
export interface AssistantNavLink { label: string; href: string; }
```
Роутер `assistant.ts` при отдаче на фронт валидирует `href` по карте (defense-in-depth) и хранит `category` + (по-прежнему) `inDomain`.

### Карточка (`apps/web/src/components/assistant/AssistantCards.tsx`)
Новый вариант рядом с урок/задача: навигационная строка — иконка + `label` + шеврон. Клик = `router.push(href)` (клиентская навигация, тот же таб). Drawer при переходе закрывается; нить диалога персистится на сервере (одна на юзера) → переоткрыл drawer, история на месте. **propose→click строго:** карточка только ведёт на страницу; действие (отменить подписку и т.п.) делает сам юзер там. Агент не автодействует.

`getConversation` (обогащение истории) отдаёт `navLinks` наравне с уроками/задачами, чтобы карточки восстанавливались при переоткрытии.

## 7. Тесты (TDD), калибровка, CI-гард

**Юнит (Vitest):**
- Гейт возвращает верную `category` на фикстурах (по паре примеров на каждую из 4 категорий; fail-open → `material`).
- Матчер карты: корректные top-K записи на релевантном запросе; узкий вопрос → 1 запись, широкий → несколько; промах ниже порога.
- Концьерж-synthesize: ответ адаптирован под вопрос, но **граунден на записях** (факты/шаги/ссылки — только из источника); **не выдумывает** шаги/href; dynamic-хит опирается на факты резолвера.
- Пайплайн роутит по категории (4 ветки).
- Whitelist: `navLinks[].href` только из карты.
- Резолвер `courseFacts`: считает опубликованные уроки, уважает `isHidden`.

**CI-гард:** тест проверяет, что каждый `deepLink.href` в карте резолвится в реальный роут приложения (мёртвые ссылки падают на сборке).

**Калибровка порога** матча — маленький eval-набор реальных вопросов (паттерн Track B intent-eval, ~20 кейсов: platform_help хиты/промахи + пара material/off_domain для проверки гейта). Гоняем, пока precision матч/промах не устроит.

**E2E** — по образцу существующих, но можно отложить (как в v1.0).

## 8. Миграции и деплой

- **Единственная миграция:** `AssistantMessage.category String?` (аддитивно, Mgmt API). Прод-коду до релиза невидима.
- Всё остальное — код + конфиг + committed JSON векторов в репо. Прод-БД для карты не касаем.
- Релиз — по-прежнему пакетом: merge `feature/ai-assistant` → master + прод-билд `--no-cache web` + флаг `ASSISTANT_ENABLED=true` включаем отдельно. Откат: revert merge (миграция аддитивна, безвредна) / не включать флаг.

## 10. Риски и поддержание карты

- **Дрейф карты** (UI поменялся → агент врёт): конфиг лежит в репо рядом с роутами → правится в том же PR, что меняет UI; CI-гард на мёртвые href; ревизия карты на каждом релизе, меняющем навигацию; FAQ и концьерж на одной карте.
- **LLM-синтез искажает шаги** (v1 адаптирует под вопрос, не отдаёт дословно) — митигация: жёсткий промпт «только факты из предоставленных записей, не добавлять шагов/кнопок/ссылок», whitelist href, низкая температура, eval-набор с проверкой отсутствия выдуманных действий.
- **Большая карта → шире поверхность дрейфа** — чем полнее карта, тем больше записей устаревают при редизайне. Митигация: карта строится/сверяется повторяемым аудитом кодовой базы, CI-гард на мёртвые href, ревизия на релизах с изменением UI.
- **Порог матча** — слишком высокий даёт много ложных промахов (агент кажется бесполезным), слишком низкий — уверенные ответы не по адресу. Калибруется eval-набором до релиза.
