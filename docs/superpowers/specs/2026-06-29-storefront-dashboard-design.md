# Витрина (storefront) на `/dashboard` — дизайн

**Дата:** 2026-06-29
**Статус:** утверждён owner в брейншторме, готов к плану
**Задача:** #3 из `docs/superpowers/plans/2026-06-26-funnel-monetization-roadmap.md`

## Проблема

Половина регистраций не открыла ни одного урока. Диагностика и поиск требуют энергии юзера → low-intent отваливается. Текущий `/dashboard` — «пара кнопок куда идти» (hero-статы + 3 бенто-входа + диагностика + след. урок + недавняя активность + радар + средний балл), он не отвечает на вопрос «что мне посмотреть прямо сейчас».

## Решение

Превратить `/dashboard` в **редакторскую витрину** — ленту «полок» с рекомендациями на основе пассивного сигнала онбординга (`marketplaces` / `goals` / `experienceLevel`) и ручных редакторских тегов. Не инструмент, а магазин: один бесплатный «ага» до любого запроса.

Сейчас `/dashboard` — реальная посадка после онбординга. `/learn` — только редирект (`→ /learn/plan` если есть трек, иначе `→ /learn/library`).

### Три компонента на одном хребте (теги)

1. **Хребет — теги.** Поле `badges String[]` на `Lesson` и `Job` (аддитивно, запрашиваемо). Первая пачка проставляется программным сидом (мы), админ-редактор для методологов — следующая фаза. `NEW` — ручной тег (у `Lesson` нет `createdAt`), поэтому `createdAt` не добавляем.
2. **Витрина-лента** на `/dashboard` — полки, карусели, страница-коллекция.
3. **Глубина** — generic страница-коллекция + возврат фильтров в Базу знаний.

## Принцип разметки (как приручаем пересечение измерений)

Контент пересекается по 4 осям: тип (джоба/урок) × задача (goal) × маркетплейс × тег. Чтобы это не превратилось в кашу:

- **Тип контента → форма карточки.** Джоба = широкая карточка (результат + «N уроков»), урок = компактная (длительность). Юзер различает глазами, не читая. Переиспользуем существующие `JobCard` и `LessonCard` как есть.
- **«Почему показали» → заголовок полки.** Полка несёт контекст: «Под задачу: Реклама», «Новое на Wildberries». Контекст не нужно парсить из карточки.
- **Тег → бейдж на карточке** (NEW / HOT / 5 мин).
- **Маркетплейс** разводим заголовком полки (если у юзера и WB, и Ozon), а не отдельной осью карточек.
- **«Развод джоб и уроков»** живёт не в лёгкой ленте (там он визуальный, по форме), а в **странице-коллекции** через переключатель `[Всё · Задачи · Уроки]` — где у юзера уже есть намерение копать.

## Таксономия тегов

Фиксированный набор `badges` (расширяемо позже):

| Тег | Полка | Кто ставит | Бейдж |
|-----|-------|-----------|-------|
| `START` | «Начни отсюда» (1–3) | методолог | — |
| `NEW` | «Новое» | методолог | NEW |
| `HOT` | «Хит платформы» | методолог сейчас / авто из завершений позже | HOT |
| `QUICK` | «Быстрые победы» | методолог (а не сырая длительность) | 5 мин |

Бесплатное/платное **не** делаем тегом — это уже показывает замок на `LessonCard` (`locked`).

## Структура `/dashboard`

### Zone 1 — Верх (остаётся, облегчённый)
- Слим-привет + 3 мини-статы (пройдено / время / стрик).
- Функц. кнопки-входы: «Мой план», «Решить задачу», «База знаний», «Диагностика» (как ускоритель, не шлюз).
- Баннер заполнения профиля (если `!profile.name`) — как сейчас.

### Zone 2 — Лента полок
Порядок:
1. **Начни отсюда** (`START`, 1–3, микс) — первый «ага».
2. **Продолжить** (`LessonProgress.status = IN_PROGRESS`) — показывать если есть.
3. **Под твою задачу: {goal}** — по одной полке на каждую цель из `UserProfile.goals` (микс джоб+уроков).
4. **Новое на {маркетплейс}** (`NEW` + фильтр marketplace; отдельная полка на каждый MP юзера).
5. **Быстрые победы** (`QUICK`).
6. **Хит платформы** (`HOT`).

### Выкинуть с `/dashboard`
- **Радар знаний** — живёт только в диагностике (решение owner).
- **Средний балл** — дубль результатов диагностики.
- **Полная «Недавняя активность»** — свернуть в «Продолжить» / убрать (слабый сигнал на витрине).

## Поведение полки (overflow)

Гибрид: **горизонтальная карусель с лимитом**.
- Полка всегда 1 строка. В карусели лимит ~12 карточек (не бесконечная подгрузка).
- **Умные стрелки:** левая отсутствует на старте, появляется только после прокрутки вправо; правая исчезает в конце. Без «колхоза».
- Заголовок полки + последняя ячейка карусели = **«Смотреть все (N) →»** → страница-коллекция.

## Страница-коллекция (новая поверхность)

Generic-шаблон, один на все полки. Маршрут: `/dashboard/collection/[shelfKey]` + query-параметры под-фильтров.

- Принимает ключ полки (`start`, `new-wb`, `goal-ads`, `hot`, `quick`…) + контекст юзера → гоняет тот же билдер, что строил полку, без лимита.
- **Группировка по типу:** секция «Задачи (N)» (сетка 3 кол) + секция «Уроки (N)» (сетка 4 кол). Разные размеры карточек не дерутся.
- **Переключатель `[Всё · Задачи · Уроки]`** фильтрует секции.
- **Чипы-фильтры:** маркетплейс (если 2) + теги (NEW/HOT/Быстрые).
- Хлебная крошка «‹ Назад на главную».

Это «пространство, которого нет» — смешанного вывода джоб+уроков под конкретного юзера на платформе сейчас не существует (есть только `/learn/solutions` = джобы, `/learn/library` = уроки).

## Возврат фильтров в Базу знаний

`FilterPanel` уже собран (категория/статус/маркетплейс/темы/сложность/длительность/курс), URL-параметры и client-side логика на месте — **просто примонтировать обратно** в `/learn/library` (отвалился в рефакторе 61-05). Плюс добавить фильтр по тегам `badges` (NEW/HOT/START/QUICK).

## Backend

### Схема (аддитивно, через Supabase Mgmt API)
- `Lesson.badges String[] @default([])`
- `Job.badges String[] @default([])`

### Билдеры (новые)
- `dashboard.getStorefront()` → массив полок; каждая `{ shelfKey, title, marketplace?, items: (Job|Lesson)[≤12], totalCount }`. Сервер фильтрует по `UserProfile.goals/marketplaces`, тегам, прогрессу.
- `dashboard.getCollection({ shelfKey, type?, marketplace?, badge? })` → `{ jobs: JobSummary[], lessons: LessonWithProgress[] }` (полный набор полки, сгруппированный по типу).

Переиспуем: `JobCard`/`LessonCard` (как есть), сетку из `JobCatalog`, `filtersFromSearchParams` из библиотеки. Каркас рекомендаций частично есть (`getRecommendedJobsFromGaps` фильтрует по marketplace + слабым осям; `job.getCatalog`).

### Маппинг goal → ось/категория (для полок «Под задачу»)
Цели онбординга (7) на 5 осей джоб:

| goal | ось/категория |
|------|---------------|
| ADS | MARKETING |
| CONTENT | CONTENT |
| ANALYTICS | ANALYTICS |
| OPERATIONS | OPERATIONS |
| FINANCE | FINANCE |
| SALES | MARKETING + ANALYTICS |
| NEW_MARKETPLACE | контент с тегом `START` / новичковый |

## Cold-start / пустые состояния
- Нет `goals` → пропускаем полки «Под задачу», опираемся на `START` + `HOT` + новое.
- Нет `marketplaces` → полки маркетплейса без сплита (общий пул).
- Нет прогресса → «Продолжить» скрыта.
- Пустая полка (нет контента под тег/цель) → не рендерится.

## Что НЕ входит (явный нескоуп)
- **Админ-редактор тегов** (UI в `/admin/content/lessons/[id]` + `/admin/jobs`) — следующая фаза; первая пачка тегов сидится программно.
- **Авто-расчёт `HOT`** из `LessonProgress.completedAt` — позже, когда выработаются правила (сейчас мало данных, и сырой row-count ≠ просмотр — наш баг 28.05).
- **`Lesson.createdAt`** — не нужен, `NEW` ручной.

## Фазировка реализации
Один связный фичер, но план разложить на волны:
1. **Хребет:** миграция `badges` + программный сид первой пачки тегов.
2. **Лента:** `dashboard.getStorefront` + редизайн `/dashboard` (полки, карусель, умные стрелки, выпил радара/балла).
3. **Глубина:** страница-коллекция (`dashboard.getCollection` + `/dashboard/collection/[shelfKey]`) + возврат `FilterPanel` в библиотеку с фильтром тегов.

## Деплой / риски (паттерн проекта)
- Миграция `badges` — аддитивно через **Supabase Mgmt API** (VPS без prisma; localhost dev читает ПРОД Supabase — `prisma migrate/push` нельзя).
- Ветка → TDD → code-review субагент → staging (`--no-cache web` + content-check) → master `--no-ff` → прод. Откат `git revert -m 1 <merge>`.
- Visual-only части свериться с `docs/design-system/` (продуктовая система, светлая).
