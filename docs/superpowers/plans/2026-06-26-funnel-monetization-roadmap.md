## Задача #1 — Триал / базовый доступ

### Что сломано сейчас
- Бесплатны **первые 2 урока по полю `order` внутри курса** (`packages/api/src/utils/access.ts:4,58,109`, `FREE_LESSON_THRESHOLD = 2`).
- Эта логика из мира «навигация по курсам». Продукт развёрнут на **джобы + диагностику**: уроки в джобе берутся из разных курсов с произвольным `order` (`packages/api/src/routers/job.ts:178` использует курсовой `l.order`, не позицию в джобе). Итог: диагностика рекомендует глубокий урок по слабой зоне → он залочен; в джобе почти всё залочено. **Бесплатное меряет мёртвую ось.**
- Доступ де-факто раздаётся промокодами (114/116), триал/паевол по-настоящему не работал.

### Рекомендуемая модель (pending owner OK)
**Одна валюта — «полный доступ × срок». Магазин всегда бесплатный, за таймером только просмотр контента.**

- **Бесплатно навсегда (storefront):** главная/витрина, каталог курсов + список уроков, диагностика, AI-поиск, лайки/избранное, профиль, партнёрский курс `/mpstats-tools`. Магазин, из которого не выгоняют.
- **За триалом/оплатой:** просмотр видео/текста уроков платных курсов.
- **Убрать правило «первые 2 по order».** Вместо него — авто-триал полного доступа при регистрации.
- **Один вечнобесплатный «ага»-урок** под сегмент (или первый урок каждой джобы по её порядку) — чтобы у вернувшегося low-intent было что посмотреть, а не только полистать.
- **Сегментный момент продажи** (не случайные стены): прошёл диагностику → «открой свой план»; искал → «открой это решение»; бродил → «тебе зашли эти — вот ещё».

### Решения owner (2026-06-26, ЗАФИКСИРОВАНО)
1. **Длина триала = 3 дня** при чистой регистрации (без карты). Старт фиксированный, дальше A/B. (Длина — не рычаг: месяц бесплатно дал 2 оплаты.)
2. **Без карты.** Карта на входе отпугнёт + риск возвратов от неожиданных списаний. Момент оплаты = первый упор в paywall после триала.
3. **Промокоды НЕ отменяются** — остаются маркетинговым инструментом на фазе запуска (MPSTATS раздаёт коды разной длины, напр. 10 дней). Чистая регистрация получает меньше (3 дня). Т.е. промокод = «длиннее/именной триал», чистая рега = базовый 3-дневный триал.
4. **Бесплатное после триала = первый урок каждой джобы навсегда** (вариант B). Storefront (главная/витрина/диагностика/поиск/каталог/лайки) бесплатен всегда; первый урок джобы по `jl.order` — вечнобесплатный крючок возврата. Правило «первые 2 по курсовому order» убираем полностью.
5. **Backfill = going-forward** — ретро-триалы существующим ~110 без подписки НЕ выдаём; модель действует только для новых регистраций.

### Джобы × тариф COURSE (1990) — находка 2026-06-26 (данные прода)
`access.ts:60`: COURSE-подписка открывает урок только если `lesson.courseId == купленный курс`. Из 60 опубл. джоб **52 одно-курсовые** (Реклама 17 / AI 14 / Аналитика 13 / Ozon 8), **8 много-курсовые**. Покупатель 1 курса видит: свои ~17 джоб целиком, 8 кросс-курсовых частично, ~35 чужих курсов залочены (первый урок бесплатен). **Вывод:** продукт-на-джобах естественно продаёт PLATFORM; COURSE = вход-приманка + апселл. **TODO (не сейчас):** честная метка покрытия джобы в UI («охватывает Аналитику+Рекламу; у вас открыта Реклама → Полный доступ»), чтобы частичный лок не выглядел багом. Решение по переосмыслению тарифа — отложено.

### Точки имплементации
- `packages/api/src/utils/access.ts` — заменить логику `FREE_LESSON_THRESHOLD` на проверку триал/подписки; добавить «storefront всегда открыт» и «1 free per job».

## Задача #2 — Разведение рефералки

### Проверено вживую (прод, 26.06) — атрибуция с лендинга работает
`middleware.ts:30,87` ставит куку `referral_code` (30 дней, HttpOnly, SameSite=lax) на **любой** URL с валидным `?ref=`, включая `/`. Подтверждено curl-ом: `/?ref=REF-TEST99` и `/register?ref=REF-TEST99` → ставят куку; битый шейп (`?ref=hello`) → не ставит. Код должен matchить `^[A-Z][A-Z0-9_]{0,15}-[A-Z0-9]{2,12}$` (`attribution.ts:12`).

**Значит плумбинг готов. Остаётся UI:**

1. **Админка рефералки — переключатель целевой страницы.** В генераторе ссылки выбор «на лендинг (`/`) / на регистрацию (`/register`)» → выдаёт готовую ссылку. (Сейчас `ReferralCodeBlock.tsx` всегда даёт `/register?ref=`.)
2. **Плашка на лендинге — РЕШЕНО: верхняя лента (top ribbon).** Тонкая полоса на всю ширину над `V8Header` (синий→тёмный градиент, маркет-стиль, Onest), sticky, закрывается крестиком. Копия (бренд-корректная, без точки, «вы» строчное, без «ё»):
   - user/ambassador: `🎁 Вам подарили {N} дней полного доступа · по приглашению: {label}` + кнопка `Забрать {N} дней →`.
   - Кнопка → `/register?ref=КОД` (код сохраняется).
   **Реализация:** client-компонент над `page.tsx` (читает `useSearchParams().get('ref')`, валидирует шейп, дёргает `referral.validateCode`); `trialDays` для user-кода = `null` → фолбэк **14** (как `register-form.tsx`); ambassador → кастомные дни + `label`. Существующий `register-form.tsx:92-111` — образец копии; `ReferralBanner.tsx` (для залогиненных «приведи друга») — НЕ трогать, другой компонент.
   **Охват v1:** только `?ref=` из URL (лендинг-ссылки всегда его несут). Кука-only кейс (вернулся без `?ref=`) отложен — `HttpOnly`, с клиента не читается.

## Задача #3 — Витрина + вывод из онбординга

**Зачем:** половина регистраций не открыла ни одного урока. Витрина = дверь для low-intent, который не пойдёт ни в диагностику, ни в поиск. Делает работу выбора за пользователя. Storefront бесплатен всегда (см. #1).

**Принцип:** не инструмент (диагностика/поиск требуют энергии юзера), а **редакторская витрина** на пассивном сигнале + один бесплатный «ага» до любого запроса.

**Полки (всё из существующих данных):**
- «Начни отсюда» — 1-3 ручных урока (нужен редакторский флаг `isFeatured` / конфиг полок). Это первый «ага».
- «Под твой маркетплейс» — фильтр по `UserProfile.marketplaces` (WB/Ozon).
- «Под твою цель» — по `goals` (помнить: реальный сигнал ~76 юзеров).
- «Новое» — `Lesson.createdAt desc`, бейдж «Новое».
- «Быстрые победы» — `duration ≤ 5-7 мин`, бейдж «5 минут».
- «Продолжить» — недосмотренные (`LessonProgress` partial).
- «Популярные задачи» — featured-джобы.

**Бейджи:** Новое / Хит / 5 мин / Под тебя.

**Гочи (из нашей же истории):** «Хит/Топ» **нельзя** строить на сыром `LessonProgress` row-count (row ≠ просмотр, фикс admin-аналитики 28.05) — брать **завершения** либо ставить редакторски.

**Единственная новая техника для витрины:** редакторская курация (флаг/конфиг полок в админке). Остальное уже лежит в БД.

**Вывод из онбординга:** ответы визарда (`marketplaces`/`goals`/`experienceLevel`) → дефолтная персонализация главной без диагностики. Диагностика и поиск — необязательные «ускорители» рядом, не шлюзы. NB: жёсткий шлюз — только онбординг (`(main)/layout.tsx:66`), диагностика шлюзом **не является** (это уже так). Остаётся переименовать/разжаловать диагностику в подаче (не ломать — 57% её проходят).

## Север: «петля результата» (стратегический контекст для #5/#6)

Аналитика меряет только **потребление** (смотрел/прошёл) и **деньги** — и **ни одной метрики результата** (применил ли человек знание у себя в магазине). Это корень нулевой конверсии и рефералов: и то и другое — ВЫХОДЫ из «реализованной ценности», а её мы не замыкаем.

**Решение (не делать платформу судьёй каузации — она зеркало+тренер):**
1. **Слой 1 — сделал ли действие?** Джоба даёт конкретное действие → галочка/чек-лист (+ в идеале проверяемо в данных MPSTATS: ставка изменилась). Опережающий индикатор, чисто атрибутируется.
2. **Слой 2 — что стало с метрикой?** Показать график из реальных данных селлера (#6) + **спросить человека**: «лучше / без изменений / хуже / несезон-нет товара-не делал». Атрибуцию делает человек → весь шум (сезон, сток, выключил рекламу) становится вариантом ответа, а не провалом.
3. **Слой 3 — агрегат по когортам:** прошедшие джобу X vs нет → продуктовый сигнал + маркетинговый пруф («в среднем снизили ДРР на N%»).

Таймер замера — **per-job** (задаёт методолог: реклама — дни, SEO/карточка — недели). Это и есть момент продажи: «получил результат на своих числах — продолжай».

**Предпосылки:** #5 (discovery данных) + #6 (связка идентичности Академия ↔ кабинет; смотреть, что уже даёт Phase 64 partner-entry).
