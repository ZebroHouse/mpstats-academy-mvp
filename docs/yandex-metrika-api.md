# Яндекс.Метрика — Reporting API

**Статус:** подключено 17.07.2026. Доступ на чтение статистики платформы через официальный Reporting API.

## Зачем

Трафик (визиты, уникальные посетители, просмотры, поведение) **нигде в нашей БД не хранится** — только в Метрике. Внутренние метрики (`UserActivityDay`, DAU/WAU/MAU) считают авторизованных пользователей, а не визиты. Если нужен трафик — только этот API.

## Где лежат креды

| Переменная | Где | Что это |
|---|---|---|
| `YANDEX_METRIKA_COUNTER_ID` | `MAAL/.env` | `94592073` — счётчик MPSTATS Academy |
| `YANDEX_METRIKA_OAUTH_TOKEN` | `MAAL/.env` | OAuth-токен, scope `metrika:read` (только чтение) |
| `YANDEX_METRIKA_CLIENT_ID` | `MAAL/.env` | ID OAuth-приложения (нужен только для перевыпуска токена) |

`.env` в `.gitignore` — проверено, в индексе git не числится. Плейсхолдеры — в `.env.example`. **Реальные значения в коммиты не попадают никогда.**

Токен бессрочный, привязан к Яндекс-аккаунту, под которым выдавался (не к владельцу приложения). Права — только чтение: создать/изменить/удалить счётчик им нельзя.

## 🚨 Главное: счётчик стоит на ВЕСЬ домен `mpstats.academy`

Счётчик собирает и главный сайт академии, и платформу, и другие поддомены. **Запрос без фильтра вернёт трафик всего домена, а не платформы.**

Насколько велика разница (30 дней, июль 2026):

| Что меряем | Визиты | Уники |
|---|---|---|
| Весь домен `mpstats.academy` | 10 889 | 6 947 |
| **Только платформа** | **3 299** | **2 041** |

Платформа — примерно треть трафика академии. Взять общую цифру = **завысить втрое**.

**Фильтр обязателен:**
```
filters=ym:s:startURL=*'*platform.mpstats.academy*'
```
Работают равнозначно `=*` (wildcard), `=~` (regex), `=@` (contains) — проверено, все три дают 3 299.

Фильтр отбирает сессии, **начавшиеся** на платформе. Переходы с главного сайта внутрь платформы в эту цифру не попадут — для отчётов по платформе это правильное поведение.

## Как пользоваться

Токен — **только заголовком**, query-параметр `oauth_token` в актуальных доках не описан:
```
Authorization: OAuth <token>
```

### Проверить доступ
```bash
curl -s -H "Authorization: OAuth $YANDEX_METRIKA_OAUTH_TOKEN" \
  'https://api-metrika.yandex.net/management/v1/counters'
```

### Визиты платформы по месяцам
```bash
curl -s -G 'https://api-metrika.yandex.net/stat/v1/data/bytime' \
  -H "Authorization: OAuth $YANDEX_METRIKA_OAUTH_TOKEN" \
  --data-urlencode 'ids=94592073' \
  --data-urlencode 'metrics=ym:s:visits,ym:s:users,ym:s:pageviews,ym:s:bounceRate' \
  --data-urlencode "filters=ym:s:startURL=*'*platform.mpstats.academy*'" \
  --data-urlencode 'date1=2026-01-01' \
  --data-urlencode 'date2=2026-07-16' \
  --data-urlencode 'group=month'
```

### Итог за период
```bash
curl -s -G 'https://api-metrika.yandex.net/stat/v1/data' \
  -H "Authorization: OAuth $YANDEX_METRIKA_OAUTH_TOKEN" \
  --data-urlencode 'ids=94592073' \
  --data-urlencode 'metrics=ym:s:visits,ym:s:users,ym:s:pageviews,ym:s:bounceRate,ym:s:avgVisitDurationSeconds' \
  --data-urlencode "filters=ym:s:startURL=*'*platform.mpstats.academy*'" \
  --data-urlencode 'date1=2026-06-17' --data-urlencode 'date2=2026-07-16'
```

### Разбивка по доменам (проверить, что фильтр не врёт)
```bash
curl -s -G 'https://api-metrika.yandex.net/stat/v1/data' \
  -H "Authorization: OAuth $YANDEX_METRIKA_OAUTH_TOKEN" \
  --data-urlencode 'ids=94592073' \
  --data-urlencode 'dimensions=ym:pv:URLDomain' \
  --data-urlencode 'metrics=ym:pv:pageviews,ym:pv:users' \
  --data-urlencode 'date1=2026-06-17' --data-urlencode 'date2=2026-07-16' \
  --data-urlencode 'sort=-ym:pv:pageviews'
```

### Метрики (актуальны на 07.2026)
`ym:s:visits` визиты · `ym:s:users` уники · `ym:s:pageviews` просмотры · `ym:s:bounceRate` отказы % · `ym:s:avgVisitDurationSeconds` среднее время визита.

Группировка по месяцам: `group=month` для `/bytime`, либо dimension `ym:s:datePeriodMonth` для `/data`.

## ⚠️ Источники трафика по платформе не сходятся с итогом

При реальных 3 299 визитах разбивка по источнику даёт:

| Dimension | Сумма |
|---|---|
| `ym:s:trafficSource` | 756 |
| `ym:s:lastsignTrafficSource` | 1 616 |
| `ym:s:firstTrafficSource` | 1 587 |

Перебраны все модели атрибуции (`last`, `lastsign`, `first`, `cross_device_last`, `cross_device_last_significant`) и три формы фильтра — расхождение не лечится. Причина: источник определяется не для каждой сессии, а на платформу заходят преимущественно внутренними переходами.

**Вывод: разбивку по источникам в отчёты не ставить** — она не бьётся с заголовочной цифрой. Визиты и уники при этом сходятся идеально и им можно доверять.

Если источники всё же понадобятся — это отдельная задача: либо считать долю от атрибутированных визитов и честно это подписывать, либо ставить отдельный счётчик только на платформу.

### 🚨 `reaches` ломает воронку — нужны `visits` или `users`

У каждой цели три метрики: `ym:s:goalNNNreaches` (срабатывания), `...visits` (визиты с достижением), `...users` (люди). Замер за 30 дней с фильтром по платформе:

| Шаг | reaches | visits | users |
|---|---|---|---|
| Регистрация | 228 | 198 | 191 |
| Диагностика старт | 111 | 104 | 101 |
| Диагностика финиш | **126** | 106 | 95 |
| Открыл урок | **3899** | 535 | 176 |
| Тарифы | 309 | 225 | 151 |
| Оплата | 12 | 11 | 10 |

На `reaches` финиш диагностики обгоняет старт, а «открыл урок» даёт 3899 при 3582 визитах всего — один человек открывает урок десятки раз. **Для воронки брать `visits`**: они ещё и аддитивны по дням, поэтому сумма подневных значений точно равна прямому запросу за период.

**`users` НЕ аддитивны.** Сумма дневных уников задваивает вернувшихся: за 7 дней честных уников 920, сумма дневных даёт ~1176. Уники за период можно брать только одним запросом за целое окно.

### Деньги — только из БД

Цель `platform_payment` ловит 10–12 оплат за 30 дней там, где БД знает реальное число: клиентская цель не срабатывает при блокировщиках и уходе со страницы. Выручку и число оплат считать из `Payment.status='COMPLETED'`, цель использовать только как поведенческий шаг.

### Ограничения запросов, на которые напоролись

- **20 метрик на запрос.** Трафик (3) + цели (8×2) = 19 — впритык, поэтому крон делает два запроса вместо одного.
- **`Query is too complicated`** на длинных окнах с большим числом метрик. Бэкфилл нарезается слайсами по 60 дней.
- **Часовой пояс.** Счётчик живёт в московском времени, контейнер считает в UTC. Данные это не портит: дни, которые сохраняются, приходят из самого ответа (`time_intervals`), а не с наших часов. Расхождение может сдвинуть только границы запрашиваемого окна, что покрывается перекрытием в 8 дней.

- Таблица снапшота — `MetrikaSnapshot` (`packages/db/prisma/schema.prisma`). `windowDays=1` — подневные аддитивные метрики, `windowDays=7|14|30|90` — дедуплицированные уники за окно.

- Клиент и парсинг — `apps/web/src/lib/metrika/`. Константы и ID целей — `packages/shared/src/metrika.ts`.
- Чтение — `packages/api/src/routers/admin-analytics-funnel.ts`, UI — `/admin/analytics/product-funnel`.

**Из tRPC и UI к Метрике не ходить** — только читать снапшот. Лимит 200 запросов / 5 минут выжигается одним активным админом.

## Лимиты

- 5 000 запросов/сутки на аккаунт, сброс в 00:00 GMT
- Reporting API — 200 запросов за 5 минут
- не более 3 параллельных запросов на токен, 30 req/sec с IP
- ≤10 dimensions и ≤20 metrics на запрос; `limit` по умолчанию 100, максимум 100 000
- превышение → HTTP 429

## Перевыпуск токена

1. `oauth.yandex.ru/client/new` → тип **«Для доступа к API»** (не «Для авторизации пользователей»; тип не меняется после создания).
2. «Доступ к данным» — это **поле поиска, а не список**. Напечатать `metrika` → выбрать **`metrika:read`**. По умолчанию там показаны только Диск и Директ — легко решить, что прав Метрики нет.
3. Открыть `https://oauth.yandex.ru/authorize?response_type=token&client_id=<YANDEX_METRIKA_CLIENT_ID>` **под аккаунтом с доступом к счётчику**. Токен вернётся в адресной строке. Redirect URI не нужен.
4. Записать в `MAAL/.env`. В git — никогда.

Для чтения хватает гостевого доступа «только просмотр» (Метрика → Настройки счётчика → Доступ).
