# Phase 61.1 Plan 61.1-06: DAU/WAU/MAU Active-User Analytics (Backend) Summary

Per-day active-user tracking (DAU/WAU/MAU + stickiness) for MAAL admin analytics: a standalone additive `UserActivityDay` table fed by a fire-and-forget heartbeat in `protectedProcedure`, an `admin.getActiveUserStats` tRPC procedure computing rolling distinct counts via parameterized SQL, and an idempotent approximate-history backfill script. **Migration is NOT yet applied to prod** — additive DDL is applied separately via Supabase Management API by the orchestrator.

## What Was Built

### Task 1 — Schema + migration file (commit `7ed8112`)
- New Prisma model `UserActivityDay`: `userId String` (text, matches `UserProfile.id`), `day DateTime @db.Date`, `createdAt DateTime @default(now())`, composite PK `@@id([userId, day])`, `@@index([day])`. No FK relation (cascade-safe).
- Migration file `20260604000000_add_user_activity_day/migration.sql` — additive only.
- `prisma generate` run locally so client types include the model. **No db push.**

### Task 2 — Heartbeat in protectedProcedure (commit `ee1dc98`)
- Inside the existing debounced `lastActiveAt` branch (fires at most every 5 min), also upsert today's `UserActivityDay` at UTC midnight. Idempotent no-op on conflict via composite PK. Fire-and-forget, failure-isolated.

### Task 3 — admin.getActiveUserStats (commit `2f18d0e`)
- New `adminProcedure` `getActiveUserStats({ days: 1..90, default 30 })`.
- Parameterized raw SQL over `UserActivityDay`: a `generate_series` date window + per-day rolling DISTINCT counts (dau, wau 7-day, mau 30-day), all `::int` cast (no BigInt).
- Pure helper `utils/active-user-stats.ts` (`computeStickiness`, `mapActiveUserStats`) maps raw rows → `{ series, current, previous }`. `mau=0 ⇒ stickiness 0`.
- 10-case unit test for the pure mapper.
- Robustness fix: wrapped the heartbeat upsert in a `try` so a synchronous throw (e.g. mocked prisma without the model) stays isolated from the `lastActiveAt` chain.

### Task 4 — Backfill script (commit `1ab770a`)
- `scripts/backfill-activity-days.ts` — single parameterized `INSERT...SELECT DISTINCT...UNION ON CONFLICT DO NOTHING`. `--dry-run` (COUNT only) ALWAYS wins over `--apply`; lazy `require('@prisma/client')`; `$disconnect` in finally. Additive-only (no DROP/ALTER/TRUNCATE/db push/accept-data-loss). Top comment documents the APPROXIMATE caveat + standalone SQL for Mgmt API.
- 7-case test: SQL builders, additive-only guard, dry-run vs apply.

## Return Shape (admin.getActiveUserStats)

```ts
{
  series: Array<{ date: string /*YYYY-MM-DD*/, dau: number, wau: number, mau: number, stickiness: number /*0..1*/ }>,
  current: { dau, wau, mau, stickiness },   // last (most recent) day
  previous: { dau, wau, mau },              // first day of window (trend baseline)
}
```

## Exact Migration SQL (NOT yet applied to prod)

```sql
CREATE TABLE IF NOT EXISTS "UserActivityDay" (
    "userId" text NOT NULL,
    "day" date NOT NULL,
    "createdAt" timestamptz NOT NULL DEFAULT now(),
    CONSTRAINT "UserActivityDay_pkey" PRIMARY KEY ("userId","day")
);
CREATE INDEX IF NOT EXISTS "UserActivityDay_day_idx" ON "UserActivityDay" ("day");
```

## Exact Backfill INSERT...SELECT SQL (for Supabase Management API)

```sql
INSERT INTO "UserActivityDay" ("userId", "day")
SELECT DISTINCT "userId", "day" FROM (
  SELECT "userId", ("startedAt" AT TIME ZONE 'UTC')::date AS "day"
    FROM "DiagnosticSession"
  UNION
  SELECT s."userId", (a."answeredAt" AT TIME ZONE 'UTC')::date AS "day"
    FROM "DiagnosticAnswer" a
    JOIN "DiagnosticSession" s ON s."id" = a."sessionId"
  UNION
  SELECT "userId", ("createdAt" AT TIME ZONE 'UTC')::date AS "day"
    FROM "ChatMessage"
  UNION
  SELECT "userId", ("createdAt" AT TIME ZONE 'UTC')::date AS "day"
    FROM "LessonComment"
  UNION
  SELECT "id" AS "userId", ("createdAt" AT TIME ZONE 'UTC')::date AS "day"
    FROM "UserProfile"
) src
ON CONFLICT ("userId", "day") DO NOTHING;
```

Dry-run COUNT equivalent (rows that would be inserted):

```sql
SELECT COUNT(*)::int AS would_insert FROM (
  SELECT DISTINCT "userId", "day" FROM (
    /* same UNION as above */
  ) src
) all_signals
WHERE NOT EXISTS (
  SELECT 1 FROM "UserActivityDay" u
   WHERE u."userId" = all_signals."userId" AND u."day" = all_signals."day"
);
```

> **APPROXIMATE history caveat:** the backfill reconstructs activity only from engaged-action tables. Pure lesson viewers who never took an engaged action are undercounted, so historical DAU/WAU/MAU is a lower bound. Accurate data accrues going-forward via the heartbeat.

## Deviations from Plan

**1. [Rule 1 - Robustness] Wrapped heartbeat upsert in try/catch**
- **Found during:** Task 3 (full-suite run)
- **Issue:** Existing test prisma mocks lack the `userActivityDay` model, so `ctx.prisma.userActivityDay.upsert` threw synchronously inside the `.then`, surfacing via the outer `lastActiveAt` catch (mislabeled error). In prod the model always exists, but the failure-isolation contract should hold against any synchronous throw.
- **Fix:** Wrapped the upsert call in its own `try` so a synchronous throw can never reach the `lastActiveAt` chain.
