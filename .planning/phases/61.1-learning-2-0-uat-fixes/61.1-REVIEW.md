## Summary

15 files reviewed across W1 (track→favorites), W1b (plan redesign + cleanup script), W2 (terminology), and W5 (breadcrumb navigation). The implementation is generally solid: Rules-of-Hooks compliance is confirmed across all changed components, the cleanup script correctly isolates `LessonProgress` as read-only, and the new `getTitleBySlug` procedure uses a correctly typed `never`-return error handler.

Two structural defects stand out. First, a logic bug in `plan/page.tsx` causes the «Рекомендованные задачи» block (the main new UI delivered by 61.1-02) to be silently hidden for any user whose `LearningPath` was not rebuilt into sectioned format — the `planIsEmpty` guard gates the entire non-empty branch on `isSectioned`, which was not updated when jobs were added as an independent content source. Second, the cleanup CLI has a flag-precedence ambiguity: supplying both `--dry-run` and `--apply` silently executes a real write, because `dryRun` is only used for display output and never overrides the `apply` argument passed to `cleanup()`. Given that this script targets prod Supabase, the ambiguity is elevated to Critical.

## Critical Issues

### CR-01: `--dry-run` flag does NOT prevent writes when combined with `--apply`

**File:** `scripts/cleanup-added-jobs.ts:122-140`

**Issue:** The CLI parses `--dry-run` and `--apply` independently. `cleanup()` is always called with `{ apply }` (line 134), so if both flags are present, `apply === true` and rows are written. The `dryRun` variable only controls which log message is printed (line 137-140) — it does not gate the actual DB write. A user who types `--dry-run --apply` to "preview then confirm" would silently execute the write.

Given this is a one-way irreversible operation on prod Supabase (`saecuecevicwjkpmaoot`), the inconsistency constitutes a data-loss risk.

**Fix:** Add an explicit mutual-exclusion guard, or make `--dry-run` override `--apply`:

```ts
if (dryRun && apply) {
  console.error('Error: --dry-run and --apply are mutually exclusive. Use one at a time.');
  process.exit(1);
}
// …then pass `{ apply: apply && !dryRun }` to cleanup()
```

## Warnings

### WR-01: `planIsEmpty` gates the entire non-empty branch on `isSectioned`, hiding `addedJobs` for legacy flat-format users

**File:** `apps/web/src/app/(main)/learn/plan/page.tsx:159-160`

**Issue:** The computation is:
```ts
const planIsEmpty =
  !recommendedPath || !isSectioned || (!hasDiagnosticLessons && addedJobs.length === 0);
```
If `recommendedPath.isSectioned === false` (legacy flat-format path that was never rebuilt), `planIsEmpty` is `true` unconditionally — the empty-state card is shown even if `addedJobs` is populated. The 61.1-02 SUMMARY states "планIsEmpty расширен на addedJobs.length — jobs-only план остаётся непустым", but the implementation still requires `isSectioned` before that check is even reached.

On prod there are known legacy flat-format paths (per CLAUDE.md: `test@mpstats.academy` example). After the Phase 61.1-02 code ships but before those users rebuild, their «Рекомендованные задачи» block will be invisible.

**Fix:** Compute a separate `hasAnyContent` that combines jobs and (sectioned) lessons independently:

```ts
const planIsEmpty =
  !recommendedPath ||
  (addedJobs.length === 0 && (!isSectioned || !hasDiagnosticLessons));
```
This makes a jobs-only plan show the jobs block even when `isSectioned === false`.

### WR-02: `fromJobSlug` read from `window.location.search` at render time, not via React state — breaks on client-side navigation

**File:** `apps/web/src/app/(main)/learn/[id]/page.tsx:295-302`

**Issue:** Both `fromParam` and `fromJobSlug` are computed by reading `window.location.search` synchronously at the top of the component function body:
```ts
const fromParam = typeof window !== 'undefined'
  ? new URLSearchParams(window.location.search).get('from')
  : null;
```
In Next.js App Router with soft navigation, `window.location.search` reflects the URL at the time of the initial render. If the user navigates from one lesson page to another via the Prev/Next buttons (client-side `<Link>` navigation), `window.location.search` changes but the component state **does not re-read it** — the old `from=` value persists in the breadcrumb from the first render. The official Next.js way is `useSearchParams()` from `next/navigation`, which triggers re-renders on query change.

**Fix:**
```ts
import { useSearchParams } from 'next/navigation';
// …
const searchParams = useSearchParams();
const fromParam = searchParams.get('from');
const fromJobSlug = fromParam?.startsWith('job:') ? fromParam.slice(4) : null;
```
`useSearchParams()` must be placed above all early returns (Rules-of-Hooks compliance already established for this file).

### WR-03: `RecommendedJobsBlock` — `addJobMutation.isPending` is shared across concurrent single-add operations

**File:** `apps/web/src/components/diagnostic/RecommendedJobsBlock.tsx:16,104`

**Issue:** `pendingId` correctly tracks which single-card button is spinning, but `bulkPending` is also OR'd into `isAddPending` for every card:
```tsx
isAddPending={pendingId === job.id || bulkPending}
```
This part is correct. The actual issue is on `addJobMutation` itself: it is a single shared mutation instance. When `handleBulkAdd` calls `mutateAsync` in a loop (line 55), if a user also clicks a single-add button concurrently (unlikely but possible), `addJobMutation.isPending` becomes `true` for all cards because `addJobMutation` is shared. While `pendingId` tracks the individual card, `addJobMutation.isPending` is not used as the disabled gate here (good), but `setPendingId` is reset in `finally` regardless of which path called it, so a concurrent single-add after bulk starts would clear `pendingId` prematurely when the bulk `finally` fires.

This is a race condition in an edge case (bulk + single simultaneous click), not a data-loss risk, but it produces a visible UI glitch where a single card's spinner disappears mid-bulk.

**Fix:** Separate mutations for single and bulk, or use a `Set<string>` of pending IDs instead of a single `pendingId` string.

## Info

### IN-01: `learn/[id]/page.tsx` — chat message list keyed by array index

**File:** `apps/web/src/app/(main)/learn/[id]/page.tsx:189,890`

**Issue:** Both the mobile and desktop chat panels render messages with `key={idx}` (array index). If messages were ever removed from the middle of the array React would incorrectly reuse DOM. In this codebase messages are always appended, so there is no current functional bug, but the pattern is fragile.

**Fix:** Use a stable key. A simple approach: attach a `clientId` to each message at push time.
```ts
setChatMessages(prev => [...prev, { role: 'user', content: userMessage, clientId: Date.now() }]);
```
and `key={msg.clientId}`.

### IN-03: `job/[slug]/page.tsx` — `notFound()` called in a `'use client'` component without an error boundary

**File:** `apps/web/src/app/(main)/learn/job/[slug]/page.tsx:23`

**Issue:** `notFound()` from `next/navigation` in a Client Component throws an exception. In Next.js App Router, `notFound()` in a server component is intercepted by the nearest `not-found.tsx`, but in a client component it falls through to the nearest `error.tsx`. If neither exists for this route segment, the user sees an unhandled error page rather than a graceful 404. This was pre-existing before 61.1 (the component was already a client component due to tRPC), but the 61.1-01 change is the most recent editor of this file and the pattern is worth flagging.

**Fix:** Add a `(main)/learn/not-found.tsx` (or `job/not-found.tsx`) to handle this case gracefully, or convert the `notFound()` call to an explicit conditional render that returns a "Задача не найдена" UI with a back-link.
